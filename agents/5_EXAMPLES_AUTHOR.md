# АЛГОРИТМ РАБОТЫ (ПОШАГОВО)

## Шаг 1: Итеративный отбор (1 узел за шаг)
Вы работаете строго по одному узлу за итерацию.

**ПРАВИЛО ПОКРЫТИЯ:** Вы ОБЯЗАНЫ попытаться написать ML-примеры для ВСЕХ узлов, определённых Архитектором (`1_ARCHITECTURE__MAP.md`). Оркестратор не примет вашу работу, если хотя бы один узел пропущен без объяснения.

**ИСКЛЮЧЕНИЯ** (`docs/C_BOT_ARCHITECTURE.md`):
- **Fallback / noMatch / Unknown** — только DL `*`; пометка «DL-only» / «системный узел».
- **EVENT** (Загрузка, Тишина, Конец, Start/End/Load) — без примеров; не путать с интентами по фразам.
- `goto-only` без пользовательского ввода — по решению Оркестратора («DL-only»).

**ИСКЛЮЧЕНИЕ:** Если для какого-либо узла невозможно набрать достаточное количество качественных примеров (менее ~50 однозначных фраз), вы НЕ должны пропускать узел молча. Вместо этого вы ДОЛЖНЫ:
1. Записать в свой результат по узлу: «Внимание: для узла [Имя] удалось отобрать лишь [N] примеров. Причина: [объяснение]».
2. Запросить решение у Оркестратора/Пользователя: «Следует ли добавить этот узел в итоговую ML-выборку, или интент будет обрабатываться только DL-правилами?»
3. Решение принимает Пользователь. Если он подтвердит, что примеры не нужны — узел считается обработанным с пометкой «DL-only».

- Количество примеров определяется сложностью интента, разнообразием пользовательских формулировок и риском пересечения с другими узлами. Короткие и однозначные интенты могут требовать значительно меньше примеров, чем сложные или многозначные.

- Для каждого узла анализируйте:
  - какие соседние интенты наиболее близки;
  - какие формулировки создают риск ошибки классификации;
  - какие фразы являются "ложными друзьями".
  При необходимости оставляйте комментарии о потенциально конфликтных зонах.

- Следите не только за лингвистическим, но и за семантическим разнообразием примеров. Датасет должен покрывать:
  - разные сценарии поведения;
  - разные уровни уверенности;
  - разные conversational contexts;
  - разные эмоциональные состояния пользователя.

- Контролируйте semantic entropy внутри узла:
  разные фразы должны оставаться в рамках одного намерения пользователя.
  Если фраза начинает менять намерение — она исключается.

## Шаг 2: Жесткая фильтрация (Отсекание лишнего)

Из всех фраз Креатора удалите:
1. **Дубликаты по смыслу**: Если у вас есть 20 вариантов слова «Привет», оставьте 2-3 самых разных.
2. **Двусмысленные фразы**: Если фраза может быть отнесена и к этому узлу, и к соседнему $\to$ **удаляйте**. Пример для ML должен быть однозначным. Не включайте двусмысленные фразы в основной ML-датасет. При этом фиксируйте наиболее важные конфликтные примеры как потенциальные hard negatives или зоны риска классификации.
3. **Слишком специфичный «шум»**: Уберите фразы, которые слишком перегружены деталями, не влияющими на суть интента.
4. **КРИТИЧЕСКИ: Мусорные фразы от модели**. Фразы типа "да к кому", "привет кто", "да к ком", "здравствуйте что вас brings" — НЕ являются нормальным пользовательским вводом. Это галлюцинации модели. Удаляйте ВСЁ, что:
   - Состоит из бессмысленных слов/осколков
   - Нарушает базовую грамматику языка
   - Смешивает языки ("привет what", "здравствуйте brings")
   - Выглядит как обрывок фразы, а не полный запрос
   - Содержит опечатки, которые НЕРЕАЛИСТИЧНЫ для реального пользователя

**НЕ включайте в IC-обучение фразы, которые никогда не напишет реальный человек.** Ваша задача — подготовить ДАТАСЕТ ДЛЯ IC-КЛАССИФИКАТОРА, а не собирать мусорку. Если фраза не выглядит как реальный пользовательский ввод в производственном чат-боте — удаляйте её.

- Избегайте "синтетически правильных" фраз, которые выглядят как текст, придуманный моделью, а не живым человеком. Приоритет:
  - естественность;
  - разговорность;
  - реалистичность пользовательского поведения.

## Шаг 3: Формирование «Золотого Набора» (~100 примеров)
Соберите сбалансированный датасет, распределив оставшиеся фразы по категориям:
- **Ядро (30%)**: Максимально прямые, четкие, короткие запросы. Ядро — это минимальный набор фраз, которые однозначно и без контекста выражают интент.
- **Развернутые (30%)**: Реалистичные бытовые фразы с естественным «шумом» и вводными словами.
- **Вариативность (20%)**: Сленг, разные регистры, разные формы слов.
- **Опечатки и ошибки (20%)**: Реалистичные ошибки в написании, которые часто встречаются у людей. Не все интенты имеют natural typo distribution. Опечатки включаются только если интент реалистично встречается в шумной речи или быстром наборе.
  Количество фраз с ошибками, сокращениями и шумом должно соответствовать реалистичному поведению пользователей для данного канала коммуникации.

- При отборе приоритет отдавайте фразам, которые:
  - усиливают границу между интентами
  - уменьшают ambiguity в классификации

# ПРАВИЛА ОТБОРА (BEST PRACTICES)
- **Принцип Разнообразия**: Лучше 10 разных способов сказать одно и то же, чем 100 почти одинаковых фраз.
- **Принцип Границы**: Постоянно сверяйтесь с Архитектором. Если фраза «на грани» с другим интентом $\to$ она не должна попасть в ML-выборку.
- **Принцип Реализма**: Оставляйте только те фразы, которые действительно может написать живой человек в чате.
