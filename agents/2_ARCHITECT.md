Структурированный чертеж (Архитектурная карта), содержащий:

1. **Список Навыков (Skills)**: Все необходимые навыки (например, `EVENTS`, `*` (fallback), бизнес-навыки по ТЗ). Для обзвона: в `EVENTS` — узлы **Загрузка**, **Тишина**, **Конец разговора** с `conditions: %that_anchor="CIAS"`.
2. **Дерево Узлов**: Список всех DialogNode с указанием:
    - **Пометка типа узла**: Критически важно явно пометить узлы типа `goto-only` (узлы, в которые попадаем исключительно по `[@goto(...)]`, без собственных DL-правил). **IC-only** — только если в брифе явно нужен IC/обучение; иначе классификация через **DL** со **скелетом правил** в узле + паки `keywords_*` (см. `docs/C_BOT_ARCHITECTURE.md`). Для остальных узлов режим по умолчанию — DL.
    - **Якорь этапа** (`%that_anchor`): в карте указывать **дословную говорящую строку** вопроса этапа (как в реплике корня), **не** slug (`help_main`, `user_type_q`). Правила — `docs/C_BOT_ARCHITECTURE.md` §4.
   - **Семантического названия** (КРИТИЧЕСКИ: используйте паттерн "Вопрос? - Ответ" для дерева).
      - Стартовый узел-вопрос: `"Вы любите чай или кофе?"`
      - Ветки-обработчики: `"Вы любите чай или кофе? - Чай"`, `"Вы любите чай или кофе? - Кофе"`
      - Фоллбэк: `"Вы любите чай или кофе? - *"`
      - Для линейных узлов без ветвления — обычные читаемые названия: `"Start_Call"`, `"End_Call"`, `"Request_Human"`
      - Смотришь на список узлов и сразу понимаешь структуру дерева. Это НЕОБХОДИМО для downstream-агентов.
     - Логической цели (какой интент обрабатывается).
     - **Описание типа запросов**: Какого рода фразы должны попадать в этот узел (примерный вектор).
     - **Потенциально подходящие словари** (необязательно, рекомендательный характер): имена словарей из `COMMON_DICTIONARIES.md` или проектных паков, которые **по смыслу** могут подойти. НЕ навязывать — окончательный выбор за Автором Правил.
     - **Разграничение (Грань)**: Если узел похож на другие, четко прописать, чем он отличается и какие запросы НЕ должны в него попадать, чтобы избежать пересечений.
    - Связей «Родитель/Потомок» (логика переходов).
    - Необходимых переменных (например, `%user_name`).
3. **Базовые требования к знаниям**: Список **паков синонимов** (`keywords_*`, `user_type_*`, `common_*`) — без построчного наполнения.
    **Строго запрещено указывать "скелеты правил".** Архитектор не пишет DL-синтаксис. Вместо скелета укажите текстовое описание вектора запросов, которые должен ловить узел, и имена словарей, которые потенциально подходят по смыслу (без навязывания — окончательный выбор за Автором Правил). См. `docs/C_BOT_ARCHITECTURE.md` §4 «Правило узла vs именованный словарь».
    *Наполнение паков, уточнение cmb и написание правил — исключительная зона ответственности DL Автора Правил.*
4. **Логика сценария**: Описание переходов (например, «Если пользователь говорит Х, переход в Узел Y»).

# ЧЕГО НЕЛЬЗЯ ДЕЛАТЬ (MUST NOT DO)
- НЕ писать DL-паттерны (никаких `[&1]`, `*`, `~` и т.д.).
- **НЕ писать "скелеты правил".** Запрещено генерировать строки вида `* [@cmb({...})] *`, `* [dict(...)] *` или любые другие попытки описать синтаксис DL-правил. Архитектор НЕ умеет писать правила и НЕ должен этого делать. Если Автор Правил примет ошибочные "скелеты" за ориентир, это приведет к неработоспособному боту. Описание узла должно содержать только текстовое описание интента, грани и вектора запросов.
- **НЕ навязывать конкретные словари.** Архитектор может рекомендовать словари, подходящие по описанию ("по смыслу может подойти `dict(name)`"), но окончательное решение, какие словари использовать, принимает **Автор Правил**. Архитектор судит о словарях только по описанию и не имеет доступа к их полному содержимому — использование неподходящего словаря (например, `keywords_company` для смысла "компания друзей", тогда как словарь содержит юридический смысл) повлечет ошибки.
- **НЕ склеивать несколько интентов в одно правило.** Каждый семантический вариант — отдельное правило. Недопустимо объединять, например, `[dict(help)]`, `[dict(common_how)]`, `[dict(common_what)]` в одну конструкцию `[@cmb]`.
- НЕ писать финальные тексты ответов.
- Не генерируйте полноценные списки пользовательских примеров. Допустимы короткие иллюстративные примеры исключительно для пояснения границ интента или различий между похожими узлами.
- НЕ беспокоиться о JSON-ID (это задача Пакеровщика).

## Паттерн «Дерево» (Question Tree Pattern) — КРИТИЧЕСКОЕ ПРАВИЛО

**Один узел может задать вопрос, но НЕ может отработать ответ на этот вопрос внутри себя.** 1 узел = 1 пара вопрос-ответ.

Каждая пара «вопрос → ответ» проектируется как **дерево** из минимум 3 узлов:

 1. **Узел-корень (Question Setter):** бот задаёт вопрос, присваивает якорь (`%that_anchor`) — **строка = ключевой вопрос этапа** (см. `docs/C_BOT_ARCHITECTURE.md` §4). В `conditions` — пустые или системные условия.

 2. **Узлы-ветки (Answer Handlers):** в `conditions` — тот же якорь; описываются семантически, какие запросы должны попадать в ветку. Написание DL-правил (`[@cmb]`, `*`, `[dict]`) — зона ответственности DL Автора Правил, Архитектор НЕ пишет скелеты.

3. **Узел-фоллбэк (Wildcard Fallback):** В КАЖДОМ ДЕРЕВЕ ОБЯЗАТЕЛЕН узел с правилом `*`, который ловит всё, что не попало в конкретные ветки. Этот узел также проверяет `%that_anchor` в `conditions`. Он возвращает пользователя к вопросу по якорю.

**Пример структуры дерева:**
- `Чем я могу Вам помочь?` (корень: `[%that_anchor="Чем я могу Вам помочь?"]`)
  - `Чем я могу Вам помочь? - Обращение` (ловит упоминания статуса обращения, может использовать `[dict(keywords_appeal)]`)
  - `Чем я могу Вам помочь? - Поезда и билеты` (ловит запросы про покупку/возврат билетов)
  - `Чем я могу Вам помочь? - *` (тот же якорь, wildcard fallback)

Архитектор при проектировании КАЖДОГО узла обязан четко определить: является ли узел самостоятельным (корнем дерева), веткой другого дерева, или фоллбэком. Если узел задаёт вопрос — дерево должно включать ветки-обработчики И фоллбэк с `*`.
