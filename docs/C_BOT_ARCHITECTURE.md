# C. АРХИТЕКТУРА БОТА (BOT_ARCHITECTURE)

**SSOT для:** Структура бота, навыки, узлы, EVENT, CIAS, якоря, паттерн "Задаватель vs Обработчик", naming, диалоговые паттерны, границы навыков, модель приоритетов.

**НЕ содержит:** Синтаксис DSL (ссылка → A), Runtime-ограничения (ссылка → B), Правила генерации ответов (ссылка → D), API (ссылка → F).

## 1. Типология узлов

| Тип | Назначение |
|-----|------------|
| **Контроллер** | Объединяет условие, логику (Skill) и ответ |
| **Контекстный** | `conditions` (`%that_anchor`) для обработки по-разному в зависимости от этапа |
| **Динамический** | Сложная логика в блоке ответов (`--if`, `--switch`) |
| **Композитный** | `[ExtendAnswer("...")]` для общих текстов |

## 2. Точки входа в сценарий

| Тип | Описание |
|-----|----------|
| **EVENT** | Системные триггеры в `dl_rules`: `EVENT <uuid>`. Поле JSON `events` обычно пустое. |
| **Приветственные** | Узлы, реагирующие на приветствия |
| **Триггерные** | Любой узел с правилами/примерами может стать точкой входа |
| **Fallback** | Узел с `*`. Обязателен для предотвращения молчания |

**КРИТИЧЕСКОЕ ПРАВИЛО:** Fallback (`*`) обязателен для **ВСЕХ** ботов без исключения. Даже для минимального бота из 1-2 узлов Fallback должен присутствовать — это защита от непредсказуемого ввода пользователя. Никогда не маркировать Fallback как «не требуется» в брифе, ТЗ или архитектурной карте.

**Инцидент (2026-05-28):** Оркестратор в брифе `greeting_bot` написал `Fallback: Не требуется (простой бот)`, что нарушило базовое архитектурное требование. Пользователь откорректировал.

### Обязательные EVENT для всех ботов

| UUID | Назначение |
|------|------------|
| `00b2fcbe-f27f-437b-a0d5-91072d840ed3` | Загрузка (старт / приветствие) |

### Обязательные EVENT для голосовых ботов 

| UUID | Назначение |
|------|------------|
| `00b2fcbe-f27f-437b-a0d5-91072d840ed3` | Загрузка (старт / приветствие) |
| `903f9438-5871-4160-80e0-32f418ca1169` | Тишина |
| `3f6b6289-4cf5-48da-b42e-6318899bbdd8` | Конец разговора |

EVENT-узлы в навыке `EVENTS` с условием `%that_anchor="CIAS"`.

**КРИТИЧЕСКОЕ ПРАВИЛО:** Узел Загрузки (`EVENT 00b2fcbe-f27f-437b-a0d5-91072d840ed3`) обязателен для **ВСЕХ ботов без исключения** — и для голосовых, и для текстовых (чат). Из ответа Загрузки делается `[@goto("Название узла-задавателя")]`. Загрузка — первая точка входа любого бота. Ни узел `Start_Greeting`, ни узел-задаватель, ни глобальный fallback не являются первой точкой входа.

**Инцидент (2026-05-29, Excuse Gen Bot):** Архитектор написал, что EVENT не требуется для текстовых ботов, и сделал `Start_Greeting` первой точкой входа. Это неверно. EVENT Загрузки обязателен для любого бота.

### Global Fallback vs CIAS

Глобальный fallback (`*`) **НЕ использует** `%that_anchor="CIAS"`. CIAS — для EVENT-узлов (также может использоваться в ветках дерева под якорем that_anchor или last_mark, если есть задача сделать дерево, из которого невозможно выйти, не ответил на вопрос в ветке). Fallback работает на самом низком уровне приоритета и не привязан к якорю.

**Инцидент (2026-05-29, Excuse Gen Bot):** Архитектор написал `%that_anchor="CIAS"` для глобального fallback-узла `Start_Greeting`. Это неверно — CIAS не используем в глобальном fallback.

### Переход из EVENT в Question Setter — КРИТИЧЕСКОЕ ПРАВИЛО

EVENT-узлы (Загрузка, Тишина, Конец разговора) кратковременны. Чтобы перевести пользователя в устойчивое состояние ожидания ответа, необходим явный переход в узел-задаватель (Question Setter).

**В карте архитектуры (`1_ARCHITECTURE__MAP.md`):**
- В колонке «Переходы» запрещены абстрактные стрелки (`→ G1 (якорь)`).
- Обязательно указывать команду перехода: `[@goto("Название узла-задавателя")]`.
- Узел-задаватель (Question Setter) должен быть спроектирован как **целевая точка перехода**, а не как текстовый блок.

**Почему:** Абстрактная стрелка не содержит информации о том, КАК реализовать переход в DSL. Автор Ответов не должен гадать — должен видеть готовую команду `@goto`.

**Пример корректной записи:**
- ❌ `→ G1 (якорь)`
- ✅ `[@goto("Для кого нужно поздравление?")]`

**Инцидент (2026-05-28, Greeting Bot):** Архитектор написал в карте `→ G1 (якорь)`, что привело к тому, что Автор Ответов не использовал `@goto` в EVENT-узле, и якорь не устанавливался. Якорь должен устанавливаться в целевом узле-задавателе, а не в EVENT..

## 3. ФАТАЛЬНО: Задаватель vs Обработчик

**Один узел НЕ может задать вопрос и отработать ответ!**

`(синтаксис сущностей и условий см. A_LANGUAGE_SPEC.md, runtime lifecycle см. B_RUNTIME_MODEL.md)`

`[if($да)]` проверяет сущность в ТЕКУЩЕМ запросе. В узле-задавателе пользователь ещё НЕ СКАЗАЛ ответ.

**Правильный архитектурный паттерн:**
1. **Узел-задаватель:** задаёт вопрос, устанавливает якорь `[%that_anchor="имя"]`.
2. **Узлы-обработчики:** проверяют `%that_anchor="имя"` в `conditions` + сущность в правиле.

**Принцип:** Каждая реплика бота — это либо начало ожидания (якорь), либо обработка ожидания (условие).

## 4. Иерархия и структура (Навыки → Узлы)

### Навыки (Skill)
- **Регистрация:** JSON-файлы в `Skill` — только метаданные (ID, имя, статус).
- **Реализация:** Логика вынесена на бэкенд. БЗ хранит ссылку (UUID).
- **Классификация:**
  - Инфраструктурные: `start`, `events`
  - Бизнес-сценарии: по пунктам ТЗ
  - Диалоговые: `inf_dialog`
  - Песочница: `__TEST`

**Размер навыка:** 5–15 узлов. Клиент не обязан проходить навык линейно.

### Интеграция навыков
- Поток: `DialogNode` → `skill_id` → [Бэкенд] → Результат → `DialogNode`

### Управление контентом (`Answer`)
- Полный декуплинг: тексты вынесены в отдельные JSON. `DialogNode` ссылается на UUID.
- Блочная структура: массив `blocks`.

## 5. Управление состоянием и контекстом

### Обычный якорь (`%that_anchor`)
Машина состояний: в ответе задаём этап, в `conditions` — тот же текст.

**Именование якорей (ОБЯЗАТЕЛЬНО):**

| Правило | Пояснение |
|---------|-----------|
| **Говорящий текст** | Значение якоря = формулировка вопроса этапа |
| **Совпадение с репликой** | Текст в якорях — побайтно одинаковый (регистр, пунктуация) |
| **Уникальность** | Если фраза встречается в разных контекстах — удлиняют уникальной частью |
| **Исключение** | Служебный супер-якорь **`CIAS`** — только EVENT / инфраструктура |
| **Запрещено** | `help_main`, `user_type_q`, `hotline_offer` и т.п. |

### Супер-якорь `CIAS`
Служебный супер-якорь для EVENT и глобальной инфраструктуры. Имеет приоритет над обычным якорем: если EVENT сработал при активном `%that_anchor="Укажите дату"`, узел с `CIAS` всё равно выиграет у fallback `*`.

### `%last_mark`
Мягкий контекст (последняя тема). В `conditions`: `%last_mark="текст"`. В ответе: `[%last_mark="текст"]`. Используется для тематических уточнений без жесткой смены состояния.

### `[disableautovars]`
Пересохраняет якоря и метки на **текущем** шаге. Используется в тишине, fallback, повторе для продления жизни контекста без перехода на новый этап.
`(runtime behavior см. B_RUNTIME_MODEL.md)`

### `[nothing]`
Разрешает пустую реплику на уровне движка. Без него узел с пустым ответом пропускается. Используется в технологических узлах, которые не должны выдавать речи, но должны выполнить присваивания.
`(runtime behavior см. B_RUNTIME_MODEL.md)`

### `voice.end_call`
Переменная сессии. `[%voice.end_call="True"]` — сигнал АТС на завершение звонка.
`(runtime behavior см. B_RUNTIME_MODEL.md)`

## 6. Проектирование данных: Naming Conventions

### Стандарты именования

| Сущность | Соглашение | Запрещено |
|----------|------------|-----------|
| **Словари** | `[префикс]_[категория]_[суть]` | — |
| **Переменные** | Короткие, семантические, **без префикса бота** | `%fpk_*`, избыточные префиксы |
| **Паки синонимов** | `keywords_<тема>` — однословные стеммы | Словарь как единственное правило |
| **Тип звонящего** | `user_type_physical`, `user_type_legal` | Весь матчинг в словаре без обёрток |
| **Согласие/отказ** | `common_yes`, `common_no`, `otkaz` (`reference/COMMON_DICTIONARIES.md`) | Новые `yes_dict` / `no_dict` |
| **Якоря** | Текст вопроса | `help_main`, `email_more`, `*_q` |

## 7. Архитектура Лексикона

| Слой | Где | Принцип |
|------|-----|---------|
| **Скелет матчинга** | `dl_rules` узла | Паттерны (`*`, `[@cmb]`, `{термин~}`, инлайн-синонимы, ссылки на паки) |
| **Пак синонимов** | `## Словарь:` | Стеммы строками. Без конструкции матчинга. |
| **Системные паки** | `common_yes`, `common_no`, `otkaz` | Готовые инфраструктурные словари. В узле только ссылка `[dict(...)]`. |

`(синтаксис скелета и словарей см. A_LANGUAGE_SPEC.md)`

**Когда выносить в словарь:** синонимов > 3-5 в позиции ИЛИ пакет повторяется в нескольких узлах (архитектура общего лексикона для атомарного обновления).
**Архитектурный запрет:** Переносить правила с конструкциями матчинга (`*`, `[@cmb]`) в словарь. Словарь — это только лексика.
**Допустимо:** правило вида `* [dict(name)] *` — скелет (`*`) присутствует в правиле.

## 8. Диалоговые архитектурные паттерны

### Ask → Process → Confirm
Классический цикл сбора данных:
1. **Ask:** Узел-задаватель ставит `%that_anchor="ask_data"`.
2. **Process:** Обработчик ловит `%that_anchor="ask_data"`, сохраняет сущность в переменную.
3. **Confirm:** Узел подтверждает данные пользователю, ставит `%that_anchor="confirm_data"`.
4. **Finalize:** Обработчик подтверждения проверяет `[if($да)]` → завершение этапа.

### Retry Pattern
Обработка ошибок ввода без циклических навигаций:
1. **Processor:** Не сработали условия успеха?
2. **Retry Node:** Отвечает «Не понял», продлевает якорь (`[disableautovars]`) или возвращает к `[%that_anchor="ask_data"]`.
3. **Counter:** Используется `%auto_request_counter` или локальная `%retry_cnt`. При достижении лимита → Escalation.

### Escalation Pattern
Перевод с бота на оператора при исчерпании ресурсов:
1. **Threshold:** Срабатывает при `retry_count >= N` или явном запросе пользователя.
2. **Context Transfer:** Собирает контекст в системные переменные перед переходом.
3. **Exit:** Узел-переход с `[%voice.end_call="True"]` или вызовом Skill-а передачи.

### Reusable Subdialog
Участок диалога, который может применяться в разных точках (например, "Сбор телефона"):
1. **Entry:** Любой узел ставит `%that_anchor="sub_tel_start"`.
2. **Logic:** Самодостаточный набор узлов, работающих в контексте `sub_tel_start`.
3. **Exit:** По завершении ставит `%that_anchor="parent_anchor"` и использует `[@goto]` или естественный переход.

### Goto-only Node
Узел, не имеющий ответов, но имеющий правила перехода (`[@goto]`). Используется как архитектурный мультиплексор для объединения веток в единую точку перехода.

### Silent Technical Node
Узел, не выдающий речи пользователю (`[nothing]`), но выполняющий системные действия:
- Установка скрытых переменных.
- Смена статуса.
- Вызов API без фидбека.
- Подготовка контекста для следующего реального ответа.

### Contextual Stage Processing
Использование `%last_mark` для обработки уточнений внутри этапа:
- Бот помнит последнюю тему (`%last_mark="тема"`).
- При вопросе «А сколько это стоит?» бот обрабатывает его в контексте `%last_mark`, не меняя жесткого `%that_anchor`.

## 9. Границы навыков (Skill Boundaries)

### Criteria for Splitting a Skill
Навык должен обладать высокой связностью (cohesion). Дробить навык нужно, если:
1. Размер превышает 15 узлов.
2. Появляется логика для разных ролей пользователей (физ/юрид) в одном потоке.
3. Сценарий требует параллельной разработки разными специалистами.

### Giant-Skill Anti-Pattern
Навык, в котором смешаны:
- Сбор контактов.
- Сбор параметров услуги.
- Обработка отказов.
- Глобальные перехватчики.
**Решение:** Выделить в `INF`, `BUSINESS`, `ERRORS`.

### Infrastructure vs Business Skills
- **Infrastructure (`START`, `EVENTS`):** Содержит глобальные точки входа, EVENT-перехватчики, глобальный fallback. Живёт вечно и не меняется за сессию.
- **Business (`SALES`, `TECH_SUPPORT`):** Содержит конкретные диалоги. Может вызываться, завершаться, возобновляться.

### Reusable Infrastructure Skills
Навыки, вынесенные в общее ядро:
- `COMMON_EVENTS`: Глобальная тишина, отключение.
- `COMMON_AUTH`: Сбор и валидация кода.
- `COMMON_CONTACTS`: Сбор email/телефона, доступный любому бизнес-навыку.

## 10. Priority Model

Формальная иерархия приоритет матчинга на одном шаге:

1. **EVENT / CIAS:** Глобальные системные перехватчики (`EVENT <uuid>`, `%that_anchor="CIAS"`). Имеют высший приоритет.
2. **Contextual Nodes:** Узлы, точно привязанные к текущему `%that_anchor` пользователя.
3. **Standard Nodes:** Узлы с правилами (`dl_rules`) или примерами IC, которые сработали по семантике/тексту.
4. **Fallback:** Узел-универсал (`*`). Срабатывает только если ничего выше не сработало.

## 11. Анти-паттерны

### last_mark vs that_anchor
- **❌** `%that_anchor` для мягких уточнений («чай или кофе?»), когда достаточно `%last_mark`.
- **✅** `%last_mark` в `conditions` + `[%last_mark="..."]` в ответе.

### Роль узла-перехода (`goto-only`)
- **❌** Считать узлы `[@goto]` "излишними".
- **✅** Узел-переход может быть архитектурной точкой: объединяет ветки, делает схему понятнее, отражает вопрос в названии.

### Принципы проектирования узлов (Атомарность)
- ✅ Одна реплика / одно намерение = один JSON-узел (Single Responsibility).
- ✅ DL-конструкции — только в `[]`, основной текст — БЕЗ скобок.
- ✅ Комментарии — отдельный block, `"inactive": true`, с `//`. `(форматирование блоков см. D_GENERATION_RULES.md)`
- ✅ Референции: `[&1]` только если в правиле есть `[-*-]` или `[-{a/b}-]`. `(синтаксис референций см. A_LANGUAGE_SPEC.md)`
- ✅ Генерировать вариативность: точные фразы + шаблоны + общие словари.
- ✅ Атомарность: `[dict(name)]` вместо явного перечисления синонимов в правиле.

### Глобальные перехватчики
Узлы с EVENT-условиями (`%that_anchor="CIAS"`) работают как глобальные перехватчики — они срабатывают вне зависимости от текущего диалогового этапа. Приоритет EVENT-узла с `CIAS` выше fallback-узла `*`.
