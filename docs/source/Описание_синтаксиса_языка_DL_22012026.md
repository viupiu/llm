## Профиль проекта фабрики (кратко)

Согласовано с `LANGUAGE_SPEC.md` (см. `docs/ROADMAPS_PROGRESS.md`):

- Уникальность ответов: **`:u:`**, **`:lu:`** (не обязательны `:uniq:` / `:luniq:` из §2 этого документа).
- Include в словарях: только **`--include(имя)`** (ровно два дефиса).
- Имена переменных: латиница, кириллица, цифры, `_`, `-`, `.`.
- Перевод на оператора (`CheckOperators`, `transfer`): **опционально**, в типовых ботах фабрики по умолчанию не используется.

### Порядок правки (фабрика LLM)

1. Изменить **проектные** `docs/*.md` (источник истины для генерации).
2. Обновить **этот** `Описание_синтаксиса_языка_DL_22012026.md` (паспорт, таблицы E, чеклисты F, E2E G — по необходимости).
3. Добавить строку в **changelog** (таблица выше) с версией `YYYYMMDD-X`.
4. Записать позицию в `docs/source/DOCX_SYNC_LOG.md` (журнал выполнения).
5. Отметить прогресс в `docs/ROADMAPS_PROGRESS.md`, если идёт роудмап.
6. **Пересобрать docx** (см. ниже).

### Пересборка docx из md

1. Открыть `docs/source/DOCX_SYNC_LOG.md` — список блоков и позиций в Word.
2. Источник: `docs/source/Описание_синтаксиса_языка_DL_22012026.md`.
3. Цель: `docs/source/Описание_синтаксиса_языка_DL_22012026.docx`.
4. Рекомендуемый порядок страниц в Word — по [навигации v2](#навигация-структура-v2) (ЧАСТЬ I → VIII); legacy-тело можно оставить в конце или переставить по LOG шага C.
5. После сборки: вычитка таблиц, чеклистов и кода в примерах (моноширинный шрифт для DSL).

### Версионирование

- **DOS-база** не меняется: `22012026`.
- **Редакция фабрики**: `YYYYMMDD` + суффикс шага (`-A`…`-H`) при крупных правках.
- В паспорте обновлять строку «редакция фабрики» и последнюю строку changelog.

### Ответственность при расхождении

Если DOS выпустил новый экспорт:

1. Положить новый файл в `docs/source/`.
2. Сравнить с текущим md (diff по разделам).
3. Перенести в проектные md только то, что нужно фабрике.
4. Сохранить паспорт, v2-навигацию, таблицы E, чеклисты F, E2E G — не затирать без ревью.

### Практические таблицы (быстрый старт)

Сводка для авторов ботов фабрики. Детали — в разделах ниже по оглавлению v2.

#### Таблица 1. Конструкции DL

| Конструкция | Где использовать | Минимальный пример | Частая ошибка |
|-------------|-------------------|-------------------|---------------|
| `*` / `**` / `~` | `dl_rules` — гибкий матч | `* оплатить *` · `пицц~` | Слишком короткое `**` → ложные срабатывания |
| `{a/b/--}` | Синонимы и пустой вариант в правиле | `{да/ага/--}` | Путать с `[dict]` — словарь отдельный файл |
| `[dict(name)]` | Правило или ответ | `[dict(city)]` | Словарь не создан в `Dictionary/` |
| `[lcdict(name)]` | Правило — понизить вес | `[lcdict(hello)] *` | Ждать другого поведения в ответе (там обычный `[dict]`) |
| `[&dict(n, norm)]` | Ответ после нормализующего словаря | `Город [&dict(city, norm)]` | `=>` в словаре, но norm не нужен по ТЗ |
| `--include(x)` | Строка **внутри** файла словаря | `--include(common_yes)` | Один дефис или `–include` (длинное тире) |
| `[- -]` + `[&1]` | Пара правило→ответ | Правило: `меня зовут [- -]` · Ответ: `[&1]` | `[&1]` без референции в правиле |
| `[@cmb({a},{b})]` | Перестановки слов | `[@cmb({хочу},{кредит})]` | Референция `[-{...}-]` **внутри** cmb — ошибка компиляции |
| `[if(%v)]{...}` | Ответ, inline | `[if(%city)]{[%city]}` | `[if(%v)]` в **том же** шаге сразу после `[%v="..."]` |
| `--if` / `--switch` | Отдельные блоки в `answers` | `--if(%var)` … `--endif` | Длинное тире `–if` вместо `--if` |
| `:u:` / `:lu:` | Уникальность ответа | `:lu: [dict(hi)]` | Писать `:uniq:`/`:luniq:` без необходимости (Legacy) |
| `[@goto("Узел")]` | Переход, пользователь в целевом ДУ | `[@goto("Опрос")]` | `[goto(...)]` без `@` и кавычек |
| `[@extend("Узел")]` | Вставка, остаёмся в исходном | `[@extend("Блок")]` | Путать с `[ExtendAnswer("...")]` |
| `[break]` / `[continue]` | В **целевом** ДУ при goto/extend | см. [таблицу goto](#описание-директив) | Ожидать другой текст при `extend`+`break` (см. truth table) |
| `[%that_anchor="..."]` | Ответ + `conditions` на дочерних | Вопрос → якорь → ветки | Использовать вместо `%last_mark` для мягких уточнений |
| `%that_anchor="CIAS"` | EVENT, тишина, конец звонка | в `conditions` EVENT-узла | Ставить на шаги опроса (только инфраструктура) |
| `[disableautovars]` | Тишина, fallback, повтор | после текста ответа | Забыть на шаге с пустым `[nothing]` |
| `[nothing]` | Намеренно пустая реплика | `[nothing]` | Пустой ответ без тега — узел пропускается |
| `[@hvs("t","r")]` | Таблицы в ответе | `[@hvs("tariff", "[&1]")]` | Имя таблицы/строки не существует в ДОС |
| `[action(n,p="v")]{...}` | Внешний вызов | см. §7 / `PATTERNS.md` | Нет `{}` для записи `[[полей]]` результата |

#### Таблица 2. Переменные

| Переменная | Назначение | Когда не использовать / осторожно |
|------------|------------|-------------------------------------|
| `%user_name` и др. пользовательские | Хранение данных сценария | Без карточки в разделе «Переменные» DOS |
| `%that_anchor` | Жёсткий этап сценария | Для «чай или кофе?» — лучше `%last_mark` |
| `%last_mark` | Мягкий контекст уточнения | Как замена полноценному якорю этапа |
| `%reply-` | Повтор последней реплики бота | Voice: SSML в `%reply-` не озвучится как разметка |
| `%query-` | Эхо запроса пользователя | В проде без необходимости — шум в ответе |
| `%auto_reply_counter` | Счётчик ответов из одного ДУ | Без порога в `if` — переменная бесполезна |
| `%auto_request_counter` | Счётчик запросов подряд | То же |
| `%user_timezone` | Время/дата для `[@Date]` | Не заведена в DOS вручную |
| `%wflag_hide_reply` | Скрыть реплику в чаттере | **Channel:** web/chat, не voice |
| `%voice.end_call` | Завершить звонок | **Channel:** телефония; не для web-only |
| `$city`, `$sys.*` | Сущность из NER / системная | В `conditions` синтаксис `%var`, сущности — `$name` в if |

Имя переменной: латиница, кириллица, цифры, `_`, `-`, `.` — см. [§4 Переменные](#часть-iii--variables-and-context-переменные-и-контекст).

#### Таблица 3. Функции (часто используемые в фабрике)

Полный каталог — [ЧАСТЬ V](#часть-v--functions-функции-dl). Ниже — типовой минимум.

| Функция | Тип аргументов | Ограничения / примечания |
|---------|----------------|-------------------------|
| `[@Date()]` | — | Текстовая; подставляет дату; TZ — `%user_timezone` |
| `[@Time()]` | Текстовый (разделитель) | Текстовая; не в условии `if` без обёртки |
| `[@ProperName("...")]` | Расширенный | Текстовая; капитализация имени |
| `[@ToUpper(...)]` / `[@ToLower(...)]` | Расширенный | Текстовая; переменные в `[%...]` внутри аргумента |
| `[@Concatenate(a,b,...)]` | Расширенный | Текстовая; склейка строк/URL |
| `[@IsEqual(a,b)]` | Расширенный | **Только в `[if(...)]`** | Нельзя вне условия |
| `[@Greater(a,b)]` / `[@Less(a,b)]` | Расширенный | **Только в `[if(...)]`** | Сравнение целых |
| `[@And(a,b,...)]` | Расширенный | **Только в `[if(...)]`** | Логическое И |
| `[@SetValue("var", val)]` / `[@set(...)]` | Переменная + значение | Имя var **без** `[%]`; может менять переменную |
| `[@TimeGreater(...)]` | Расширенный | Условная; время/дата | См. NB про `00:00` в §7 |

**Правило:** текстовые функции — в теле ответа или в `{...}` после `if`; условные — **только** внутри `[if(...)]` / `[elsif(...)]`.

### Чеклисты перед релизом (шаг F)

Используйте перед импортом архива в ДОС. В фабрике LLM дублируются в `docs/VALIDATION.md` и `docs/VOICE_OUTBOUND_CHECKLIST.md`.

#### Чеклист A. Любой бот (production)

**Архив и JSON**

- [ ] Корень ZIP: `metadata.json` + папка `[ASSISTANT_ID]-master-[40hex]` (см. `GENERATION_RULES.md`)
- [ ] `metadata.json`: формат `lc_version`, не `{"assistant_id":"1.0"}`
- [ ] Все `id` — сгенерированные UUID, не заглушки `00000000-...`
- [ ] `skill_id`, словари, переменные — связи существуют в папках архива
- [ ] `created` / `metadata.timestamp` — naive ISO без `+03:00` / `Z` (как в эталоне)

**DSL в узлах**

- [ ] У fallback-узла правило `*` и осмысленный ответ (не пустой без `[nothing]`)
- [ ] `[@goto("Имя")]` / `[@extend("Имя")]` — с `@`, кавычки, имя = `DialogNode.name`
- [ ] `[&1]` только при `[- -]` / `[-{...}-]` в том же правиле
- [ ] `[dict(name)]` — файл `Dictionary/{id}.json` существует; в словаре есть канон без `=>`
- [ ] `--include(...)` только с **двумя** дефисами
- [ ] Блочные теги: `--if`, `--switch`, `--endswitch` — не длинное тире `–`
- [ ] Нет проверки `[%var]` в том же ответе сразу после присвоения `[%var=...]`
- [ ] Слоты: `required: true` → `question`, `retry_count`, `failure_answer` (JSON, не if в тексте)

**ML (если есть примеры)**

- [ ] ≥ 10 уникальных примеров на класс; ≥ 3 класса в ассистенте
- [ ] Узел только по IC: **нет** `[&1]` / референций на правила в ответе

**Перед упаковкой (фабрика)**

- [ ] `python scripts/validate_archive.py --staging work/staging`
- [ ] `python scripts/pack_archive.py ...` → повторная `validate_archive.py --zip`

#### Чеклист B. Голосовой обзвон (voice / телефония)

Дополнительно к чеклисту A. Эталоны: `archives/examples/`, `docs/PATTERNS.md` (SSML).

**Инфраструктура**

- [ ] Навык `EVENTS`: **Загрузка**, **Тишина**, **Конец разговора**
- [ ] В `dl_rules` — `EVENT` с UUID (см. `BOT_ARCHITECTURE.md`)
- [ ] У EVENT-узлов в `conditions`: `%that_anchor="CIAS"`
- [ ] Навык `*`: fallback + `[disableautovars]` где нельзя потерять этап
- [ ] Переменная `voice.end_call`; в конце: `[%voice.end_call="True"]`

**Сценарий**

- [ ] Этапы: вопрос в ответе → `[%that_anchor="..."]` → дочерние узлы с тем же якорем в `conditions`
- [ ] На этапе: варианты ответа + узел `Вопрос - *` для нераспознанного
- [ ] Пустой ответ по задумке — `[nothing]` (+ `[disableautovars]` при необходимости)
- [ ] Общие словари: `reference/COMMON_DICTIONARIES.md` — не дублировать `common_yes` / `otkaz`

**Голос**

- [ ] Прощание: при необходимости `<speak>...<break>...</speak>`, `[%voice.end_call]` **вне** `speak`
- [ ] Тишина / повтор / fallback — `[disableautovars]`
- [ ] Узел «Повтори» — `[%reply-]` при необходимости
- [ ] `CIAS` **не** на шагах бизнес-опроса, только EVENT/инфраструктура

**Не смешивать**

- [ ] EVENT только в `dl_rules`, не в JSON-поле `events`
- [ ] `=>` в словарях — только если нужен `[&dict(..., norm)]`

### Эталонные примеры (шаг G)

В legacy-разделах ниже по файлу встречаются старые формулировки DOS. **Ориентир для копирования — этот блок** и `docs/PATTERNS.md`.

#### Позитив / негатив (частые кейсы)

| Тема | Верно | Неверно |
|------|-------|---------|
| goto | `Привет! [@goto("Узел 2")] Как дела?` | `[@goto(Узел 2)]` · `[@goto("2")]` без `@` |
| extend + break | Итог: только хвост исходного (`Привет! Как дела?`) | Ожидать текст целевого ДУ в ответе |
| if + переменная | Шаг 1: `[%_x="[$city]"]` · Шаг 2: `[if(%_x)]{...}` | В одном ответе: `[%city="Москва"][if(%city)]{...}` |
| cmb + референция | Отдельные правила без `[-...-]` внутри cmb | `[@cmb({[-{да/нет}-]}, {бот})]` |
| include | `--include(common_yes)` | `–include(common_yes)` |
| уникальность | `:lu: [dict(greet)]` | `:uniq:` в новых ботах фабрики (достаточно `:u:`/`:lu:`) |

> В §3.12 (примеры 1–6) исправлены кавычки в `[@goto("2")]` / `[@extend("2")]`; смысл — см. [таблицу goto](#описание-директив).

#### Сквозной сценарий (end-to-end): выбор типа кредита

Эталон связки **правило → якорь → conditions → fallback → повтор**. Имена узлов — смысловые (как в фабрике).

**Словари (фрагмент):**

```text
mortgage_types
Ипотека
=>ипотек~
=>жиль~
Автокредит
=>автокредит~
=>машин~
```

**Узел «Кредит — вопрос»**

```text
Правила:
* [@cmb({[dict(mortgage_types)]}, {кредит/заем})] *

Ответ:
Выберите тип кредита: ипотека или автокредит?
[%that_anchor="тип кредита"] [%last_mark="тип кредита"]
```

**Узел «Кредит — ипотека»**

```text
Условия:
%that_anchor="тип кредита"

Правила:
[dict(mortgage_types)] * ипотек~ *

Ответ:
Хорошо, оформим ипотеку. [@goto("Кредит — детали ипотеки")]
```

**Узел «Кредит — автокредит»**

```text
Условия:
%that_anchor="тип кредита"

Правила:
[dict(mortgage_types)] * авт~ *

Ответ:
Оформим автокредит. [@goto("Кредит — детали авто")]
```

**Узел «Кредит — не распознано»**

```text
Условия:
%that_anchor="тип кредита"

Правила:
*

Ответ:
[if(%last_set1!="задали 1 раз")]{Вы не выбрали тип. Скажите: ипотека или автокредит? [%that_anchor="тип кредита"] [%last_mark="тип кредита"] [%last_set1="задали 1 раз"]}
[else]{Давайте начнём сначала. [@goto("Кредит — вопрос")]}
[disableautovars]
```

**Узел «Кредит — детали ипотеки»** (целевой после goto)

```text
Правила:
*

Ответ:
Расскажу условия по ипотеке… [continue]
```

Пользователь после goto остаётся в целевом ДУ; `[continue]` дописывает хвост из исходного ответа goto-узла, если он был.

# ЧАСТЬ I — Core syntax (базовый синтаксис)

**Supported** · Соответствует §1 и §6 DOS. Включает структуру узла, правила, ответы, комбинаторику `[@cmb]`.

1.  Основные возможности языка DL

## Общий вид данных

Язык DL состоит из правил, называемых «диалоговые узлы». Каждое правило описывает реакцию ассистента на определенную реплику (или класс реплик) пользователя.

Крупные группы узлов, объединенные одной темой, удобно выделять в отдельные навыки и присваивать им значимые имена.

## Структура узла

**Supported** · Карточка узла состоит из следующих разделов:

1.  Название узла.
2.  Описание узла.
3.  **Условия**.

В этом разделе при необходимости описывается условие на состояние переменных.

При выполнении условия данный узел будет рассматриваться при выборе ответа на реплику пользователя. Иначе узел будет исключен из рассмотрения.

1.  Намерения – для использования Глобальных намерений.
2.  **Правила**.

Описание на языке DL (Dialog Language) реплики или класса реплик пользователя, реакцию на которые описывает узел.

1.  Примеры – примеры запросов пользователя, при которых данный узел должен сработать, используются для обучения модели ML (Machine Learning). Требуется минимум 10 примеров в одном узле и минимум 2 узла с примерами в ассистенте.
2.  Слоты – раздел для описания поведения ассистента в случае, если пользователь дал не всю необходимую информацию.
3.  **Ответы**.

Описание на языке DL возможных реакций на реплику/класс реплик пользователя, описанных в разделе «Правила»/«Примеры» данного узла.

Если в узле несколько ответов, то при выборе этого узла как наиболее релевантного случайным образом выбирается один из ответов.

1.  Тесты – раздел для задания тестовых вопросов-ответов для автоматической проверки работы узла.

## Правила

**Supported** · В строках Правила нет разницы между прописными и строчными буквами. (Но в Ответах буквы выводятся для пользователя так, как они даны в узле.)

Правило «Москва» сработает и на запрос «Москва», и на запрос «москва».

- В Правилах нет разницы между буквами е и ё.

Правило «счет» сработает и на запрос «счет», и на запрос «счёт».

- Все небуквенные символы в запросе пользователя заменяются на пробелы, за исключением символов !, ?, @, $, #, \*, +, =, %, &, ₽, £, €. Поэтому в Правилах нет смысла описывать остальные символы.

Правило «кто-то» сработает и на запрос «кто-то», и на запрос «кто то».

### Варианты задания запроса в правилах

В правилах необходимо описывать возможные варианты реплики пользователя. Варианты можно задать несколькими способами:

- **Синонимичные строки**

Варианты можно записать в нескольких строках:

**Пример**

Как тебя зовут?

Как звать-то \*?

Как твоё имя \*?

- **Словари**

Варианты можно перечислить в отдельном словаре, а в узле поставить ссылку на имя словаря в нужном месте правила. Подробнее см. ниже [раздел «Словари»](#_Ссылка_на_словарь).

- **Inline-словари**

Синонимичные варианты можно записывать непосредственно в одной правила.

Синтаксис: {string/string}

или {string/string/--} -- обозначает пустой вариант.

**Пример**

Правило: {Скучно/Грустно/Одиноко} мне.

Сработает на «Скучно мне», «Грустно мне», «Одиноко мне»  
<br/>Правило {Я/ты/--} молодец  
Сработает при отправке «Я молодец», «Ты молодец» , «молодец»

В **угловых скобках** записывается опциональный элемент, например: &lt;!&gt;

**Пример**

Правило: Ты кто &lt;?&gt;

Сработает и на «Ты кто», и на «Ты кто?»

В угловых скобках может быть записан и словарь:

&lt;\[dict(название словаря)\]&gt;, также работает запись \[&lt;dict(название словаря)&gt;\]

### Кванторы в правилах

Кванторы служат для расширения правила, придания ему нечёткости.

Разрешены следующие виды кванторов:

- **Звездочка** \* используется для обозначения любого слова или последовательности слов. Звездочка также может означать отсутствие слова.

**Пример**

Правило: _\* адрес_

Сработает на запросы: «_какой ваш адрес_», «_дайте адрес_» и т.д., либо просто «_адрес_».

- **Суперзвездочка** \*\* означает наличие любого слова или значимого символа !, ?, @, $, #, \*, +, =, %, &, ₽, £, € или последовательности слов или значимых символов, и, в отличие от звездочки, не может равняться нулю.

Суперзвездочка при разборе узла имеет приоритет над звездочкой.

Конструкция \*\* понимается как одна суперзвездочка.

- **Тильда** ~ служит для обозначения произвольного начала и/или окончания слова, пишется вплотную к слову. Ассистент трактует слово с тильдой как некоторый набор возможных слов.

**Примеры**

Правило: кнопк~

Сработает на запросы: «кнопка», «кнопки», «кнопке» и т. д.

Привило: ~к

Сработает на любое слово, заканчивающееся на букву к.

Привило: ~регистр~

Сработает на запросы: «регистрация», «зарегистрироваться», «регистр» и т.д.

## Ответы

**Supported** · В разделе Ответы может быть несколько ответов. Ответ пользователю выбирается по случайному счетчику, если не задан другой порядок.

### Варианты задания ответа

- **Синонимичные строки**

Варианты ответа можно записать несколькими подряд идущими строками.

**Пример**

Правило: Спасибо

Ответ: Да не за что...

Ответ: Обращайтесь, если что.

Ответ: Всегда пожалуйста.

- **Словари**

Варианты ответа или части ответа можно перечислить в отдельном словаре, а в узле поставить ссылку на имя словаря в нужном месте ответа.

- **Inline-словари**

Варианты можно записывать непосредственно в ответе в узле.

Синтаксис: {string/string}

или {string/string/--} -- обозначает пустой вариант.

**Пример**

Ответ: Как {дальше/дальше с этим/после этого} жить...

### Кванторы в ответах

- **Вставка значения звездочки в ответ**

Звездочка \* означает любой набор слов в некотором месте правила. В реальном диалоге пользователь на месте, где стоит звездочка, вводит конкретные слова. Эти слова называются **реализацией звездочки**. Реализацию звездочки можно использовать для формирования ответа.

Для вставки в ответ реализации звездочки используются следующие конструкции:

\[\*\] - вставка реализации звездочки (первой слева звездочки, если их в вопросе много),

\[\*1\] – вставка первой слева реализации звездочки,

\[\*2\] – вставка второй слева реализации звездочки,

\[\*3\] – вставка третьей слева реализации звездочки.

Здесь с точки зрения вставки звездочка «\*» и суперзвездочка «\*\*» не отличаются.

**Пример**

Правило: \* получить техподдержку \*

Ответ: По вопросу \[\*2\] обращайтесь в нашу службу поддержки [support@company.ru](mailto:support@company.ru)

Также существует конструкция, которая позволяет выделить конкретную часть запроса. В правиле ту часть запроса, которую мы хотим использовать в ответе ассистента, обрамляем квадратными скобками и тире:

\[-\*-\]

\[-\*\*-\]

\[-a-\]

\[-{a/b/c}-\]

\[-\[dict(name)\]-\]

а в ответе ссылаемся на эту часть запроса следующим образом:

\[&X\]

где X – цифры, номер той части запроса поотзователя, к которой идет отсылка в ответе, считается слева направо по открывающей квадратной скобке в правиле.

\[&1\] – вставка первой слева реализации звездочки,

\[&2\] – вставка второй слева реализации звездочки,

\[&3\] – вставка третьей слева реализации звездочки.

**Пример**

Правило: \* получить техподдержку по вопросу \[-\*\*-\]

Ответ: По вопросу \[&1\] обращайтесь в нашу службу поддержки [support@company.ru](mailto:support@company.ru)

1.  Комбинаторика в правилах

**Supported** · В правиле можно использовать функцию combine, или сокращенно **cmb.** Эта функция (незримо для нас) меняет местами свои аргументы во всех комбинациях и генерирует для каждого из них строку правила, при этом между аргументами добавляется звездочка. Каждый аргумент такой функции должен быть обрамлен фигурными скобками.

**Пример**

Правило: a \[@cmb( { test \[dict(ttt)\] }, { aaa } )\]

Ответ: res

Раскроется в:

Правило: a test \[dict(ttt)\] \* aaa

Правило: a aaa \* test \[dict(ttt)\]

Ответ: res

Также можно использовать функцию combine_strict или сокращенно **cmb_s**, для предотвращения добавления звездочек между аргументами.

**Пример**

Правило: a \[@cmb_s({test \[dict(ttt)\]}, {aaa})\]

Ответ: res

Раскроется в:

Правило: a test \[dict(ttt)\] aaa

Правило: a aaa test \[dict(ttt)\]

Ответ: res

Количество аргументов функции cmb и cmb_s не ограничено.

Можно использовать несколько cmb и/или cmb_s в одном правиле.

ВАЖНО: нельзя использовать референцию внутри cmb, т.е. **нельзя** создавать правила как в примере:

Правило: \* \[@cmb({\[-{умный/веселый}-\]}, {бот} )\] \*

Ответ: Да, я \[&1\] бот.

На такое правило компиляция выдаст ошибку.

**Cmb_n** – функция, которая подставляет между аргументами число звездочек, которое указано в первом аргументе. Структура записи: \[@cmb_n(ЧИСЛО_ЗВЕЗДОЧЕК, {аргумент1}, {аргумент2}, {аргумент3})\].

Словари с комбинаторикой нельзя использовать в слотах. При обучении выдается ошибка.

# ЧАСТЬ II — Execution semantics (исполнение ответа)

**Supported** · Соответствует §2 и §3 DOS: модификаторы веса, условия, переходы, break/continue.

1.  Модификаторы приоритета ответов

> **Согласование с профилем фабрики (см. паспорт документа):**  
> - `:u:` — **сокращённая запись** модификатора `:uniq:` (один раз за сессию).  
> - `:lu:` — **сокращённая запись** модификатора `:luniq:` (один раз за цикл узла, затем сброс).  
> В `docs/LANGUAGE_SPEC.md` для генерации ботов фабрики рекомендуется писать `:u:` и `:lu:`; поведение то же, что у полных имён ниже.

## Модификатор пониженного веса :last:

**Supported** · Ответ с таким модификатором будет сгенерирован только если не получилось сгенерировать ответ среди ответов без такого модификатора во всех равновесных узлах.

**Пример**

Узел 1

Правило: qwe

Ответ: :last: 5

Узел 2

Правило: qwe

Ответ: \[if(%user_1!= "1")\] { 1 \[%user_1="1"\] }

Ответ: \[if(%user_2!= "2")\] { 2 \[%user_2="2"\] }

Ответ: :last: 3 \[%user_1=""\] \[%user_2=""\]

Ответ: :last: 4 \[%user_1=""\] \[%user_2=""\]

В вышеописанном примере ответы «3», «4» и «5» могут сгенерировать только при условии, что ранее в разговоре были сгенерированы ответы «1» и «2».

Сокращенное имя: l

Ограничения: не допускается одновременное использование в одном ответе модификаторов last и luniq.

## Модификатор одноразового ответа :uniq: (сокращение :u:)

**Legacy** (в фабрике пишите `:u:`) · Ответ с таким модификатором будет сгенерирован только один раз за сессию. Эквивалентная краткая запись в ответе: `:u:`.

**Пример**

Правило: qwe

Ответ: :uniq: ты увидишь этот ответ только один раз

Сокращенное имя: u

## Модификатор строгой уникальности ответа :luniq: (сокращение :lu:)

**Legacy** (в фабрике пишите `:lu:`) · Ответ с таким модификатором будет использован повторно только после того, как в его узле будут использованы по одному разу все остальные ответы, имеющие флаг uniq или luniq. Эквивалентная краткая запись в ответе: `:lu:`.

**Пример**

Правило: назови число от одного до пяти

Ответ: :luniq: один

Ответ: :luniq: два

Ответ: :luniq: три

Ответ: :luniq: четыре

Ответ: :luniq: пять

Сокращенное имя: lu

Ограничения: не допускается одновременное использование в одном ответе модификаторов last и luniq.

## Частичный вес ответа :+n:, :-n:

Частичный вес ответа представляется числом от -500 до +500 и по умолчанию равен нулю. При построении ответа среди узлов с одинаковым весом ответы выбираются по порядку, заданному частичным весом - от большего к меньшему.

**Пример**

Правило: посчитай от одного до пяти

Ответ: :luniq:+2: один

Ответ: :luniq:+1: два

Ответ: :luniq: три

Ответ: :luniq:-1: четыре

Ответ: :luniq:-2: пять

1.  Операторы

## Ссылка на словарь \[dict()\]

**Supported** · **Синтаксис:** \[dict(dict_name)\], где dict_name – имя существующего словаря.

**Область использования:** Правила, Ответы.

**Описание:**

Для компактного вида и удобства использования варианты вопроса и ответа можно записывать в отдельные словари, а в узлах ставить ссылку на эти словари.

**Пример**

Правило: \* \[dict(dialog-privet)\] \*

Ответ: Привет! Как жизнь, как настроение?

В словаре dialog-privet содержатся все варианты приветствий пользователя.

## Проверка \[if()\]

**Supported** · **Синтаксис:**

\[if(%var)\]{string} - проверка на существование значения у переменной

\[if(**!**%var)\]{string} - проверка на несуществование значения у переменной

\[if(%var="value")\] - проверка на совпадение с некоторым значением

\[if(%var**!**\="value")\]{string} - проверка на несовпадение с некоторым значением

Использование альтернативы \[elsif()\]{string} и \[else\]{string} является факультативным.

**Область использования:** Ответы.

**Описание:**

Если условие, наложенное на переменную, удовлетворяется, то вместо оператора IF вставляется текст, записанный в фигурных скобках.

Вместо значения переменной "value" можно вставлять референции \[&1\].

**Пример**

Правило: Как меня зовут

Ответ: \[if(%user_name="Антон")\]{ Вас зовут Антон, как и меня! } \[elsif(%user_name)\]{ Вас зовут \[%user_name\]! }\[else\]{ Я не знаю, как вас зовут. }

## Проверка \[elsif\]

**Supported** · Проверка elsif работает следующим образом: выражение elsif выполнится, если предшествующее выражение if и предшествующие выражения elsif эквивалентны FALSE, а текущий elsif равен TRUE.

**Пример**

  
Допустим, есть узлы:

Правило: вар1

Ответ: Есть вар1! \[%var1="1"\]

Правило: вар2

Ответ: Есть вар2! \[%var2="1"\]

Правило: вопрос

Ответ: \[if(%var1="1")\]{Ответ 1}\[elsif(%var2="1")\]{Ответ 2}\[else\]{Ответ 3}

В разговоре эти узлы будут работать следующим образом (в начале разговора переменные var1 и var2 - пустые):

_Пользователь:_ вопрос

_Ассистент:_ Ответ 3

_Пользователь:_ вар2

_Ассистент:_ Есть вар2!

_Пользователь:_ вопрос

_Ассистент:_ Ответ 2

_Пользователь:_ вар1

Ассистент: Есть вар1!

Пользователь: вопрос

Ассистент: Ответ 1

Иными словами:

Если **верно, что var1="1"**, то сработает Ответ 1 (**даже если при этом верно, что var2="1"**).  
Если **неверно, что var1="1", но верно, что var2="1"**, то сработает Ответ 2.

Если **неверно и var1="1", и var2="1"**, то сработает Ответ 3.

Elsif в одном ответе может быть несколько.

## Тэг \[switch\] - оператор выбора

Тэг _switch_ используется в ответе и позволяет избежать применения большого числа вложенных друг в друга тэгов _if_.

Аргумент ключевого слова _switch_ поочерёдно сравнивается с аргументами ключевых слов _case._ В случае совпадения вся конструкция тэга switch заменяется на ветку, описанную после ключевого слова _case_, точно так же, как это происходит при выборе ветки в тэге _if_. Если аргумент _switch_ не совпал ни с одним из аргументов _case_, то срабатывает ветка, находящаяся после ключевого слова _default_.

На аргументы _switch_ и _case_ накладываются такие же ограничения, как и на аргументы функций, имеющие тип _Расширенный_. Кроме того, эти аргументы не могут содержать более одного элемента языка DL, например, нельзя использовать конструкцию _\[switch( \[\*1\]\[\*2\]\[\*3\] )\]_, но можно использовать _\[switch( \[@Concatenate( "\[\*1\]", "\[\*2\]", "\[\*3\]" )\] )\]_.

Ветки _case_ и _default_ работают по тем же правилам, что и ветки тэга _if_.

**Пример 1**

Допустим, в ассистенте есть узел:

Правило: \* у меня \[-\*\*-\] глаза \*

Ответ: \[switch(\[@ToLower("\[&1\]")\])\] \[case(карие)\] { Мне нравятся тёмные глаза. } \[case(синие)\] { Красота! }\[case(зеленые)\] { Прямо как у моей кошки. } \[default\] { Ничего особенного. }

В разговоре этот узел будет работать следующим образом:

_Пользователь_: У меня карие глаза

_Ассистент:_ Мне нравятся тёмные глаза.

_Пользователь:_ У меня СИНИЕ глаза

_Ассистент:_ Красота!

_Пользователь:_ У меня зеленые глаза

_Ассистент:_ Прямо как у моей кошки

_Пользователь:_ У меня фиолетовые глаза

_Ассистент:_ Ничего особенного.

**Пример 2**

Допустим, в ассистенте есть узлы:

Узел 1:

Правило: меня зовут \[-\*\*-\]

Ответ: Привет, \[@Fltr("\[&1\]")\] \[@set("user_name", "\[@Fltr("\[&1\]")\]", "1")\]

Узел 2:

Правило: \* как \* тебя \* зовут \*

Ответ: \[switch(\[%user_name\])\]\[case(Хоттабыч)\]{Я твой тезка!}\[default\]{Меня зовут Хоттабыч.}

В разговоре эти узлы будут работать следующим образом:

_Пользователь:_ меня зовут хоттабыч

_Ассистент:_ Привет, Хоттабыч

_Пользователь:_ А как тебя зовут?

_Ассистент:_ Я твой тезка!

_Пользователь:_ Меня зовут Вася

_Ассистент:_ Привет, Вася

_Пользователь:_ А тебя как зовут?

_Ассистент:_ Меня зовут Хоттабыч.

## Условные операторы уровня узла

**Supported** · Условные операторы уровня узла предназначены для упрощения описания проверок, одинаковых для нескольких ответов внутри одного узла.

Существует два типа таких операторов - If и Switch, которые действуют аналогично одноимённым тэгам, использующимся внутри ответов. Разница заключается в том, что условные операторы уровня узла указываются между ответами и действуют сразу на один или несколько ответов, а не на часть единственного ответа.

### Оператор If

Для описания условного оператора уровня узла If применяются ключевые слова if, elsif, else и endif, которые должны указываться в отдельной строке и предваряться двумя символами "минус". Ключевые слова if и elseif должны содержать в скобках после себя условие для проверки, которая будет выполняться для всех ответов, следующих от текущего ключевого слова и до следующего ключевого слова. Описание условного оператора уровня узла If обязательно должно заканчиваться ключевыми словом endif.

**Пример**

Правило: меня зовут \[-\*\*-\]

Ответ:

\--if( \[@IsEmpty("\[%user_name\]")\] )

:lu: Приятно познакомиться, \[@Set("user_name", "\[\*1\]")\] !

:lu: Привет, \[@Set("user_name", "\[&1\]")\] !

\--elsif( \[@IsNotEqual("\[%user_name\]", "\[&1\]")\] )

:lu: Хм, а в прошлый раз вас звали \[%user_name\].

:lu: А вы ничего не путаете?

\--else

Здравствуйте, \[%user_name\], я вас помню.

\--endif

В разговоре этот узел будет работать следующим образом (при условии, что в начале разговора переменная user_name - пустая):

_Пользователь_: Меня зовут Вася

_Ассистент:_ Приятно познакомиться, Вася!

_Пользователь:_ Меня зовут Петя

_Ассистент:_ А вы ничего не путаете?

_Пользователь:_ Меня зовут Петя

_Ассистент:_ Хм, а в прошлый раз вас звали Вася.

_Пользователь:_ Меня зовут Вася

_Ассистент:_ Здравствуйте, Вася, я вас помню.

### Оператор Switch

Для описания условного оператора уровня узла Switch применяются ключевые слова switch, case, default и endswitch, которые должны указываться в отдельной строке и предваряться двумя символами "минус". Описание условного оператора уровня узла Switch обязательно должно заканчиваться ключевым словом endswitch.

Допускается использование вложенных конструкций из нескольких условных операторов уровня узла одного или разных видов.

**Пример**

Допустим, есть узлы:

Узел 1:

Правило: сегодня \[-{холодно/жарко/тепло/пасмурно/дождь/гроза/облачно/ясно}-\]

Ответ: Угу, \[&1\]… \[%weather_today="\[&1\]"\]

Узел 2:

Правило: \* взять \* мне \* с собою \* зонт \*

Ответ:

\--switch( \[%weather_today\] )

\--case( ясно )

:lu: Не стоит, сегодня днём будет ясная погода.

:lu: Обязательно возьми, чтобы не обгореть на солнце.

\--case( облачно )

Нет.

Нет, сегодня днём будет облачно, но дождя не обещают.

\--case( пасмурно )

:lu: Лучше возьми, так как обещают пасмурную погоду днём.

:lu: Будет пасмурно, но дождя не обещают, можно рискнуть.

\--case( дождь )

Обязательно возьми, сегодня днём будет дождь.

\--case( гроза )

Конечно возьми, только он тебе не поможет.

\--default

Ответ: Робот потерял связь с метеоцентром, спроси робота об этом ещё раз завтра.

\--endswitch

В разговоре эти узлы будут работать следующим образом:

_Пользователь:_ Сегодня ясно

_Ассистент:_ Угу, ясно…

_Пользователь:_ Взять ли мне с собою зонт?

_Ассистент:_ Обязательно возьми, чтобы не обгореть на солнце.

_Пользователь:_ Сегодня пасмурно

_Ассистент:_ Угу, пасмурно…

_Пользователь:_ Взять ли мне с собою зонт?

_Ассистент:_ Лучше возьми, так как обещают пасмурную погоду днём.

_Пользователь:_ Взять ли мне с собою зонт?

_Ассистент:_ Будет пасмурно, но дождя не обещают, можно рискнуть.

_Пользователь:_ Сегодня холодно

Ассистент: Угу, холодно…

Пользователь: Взять ли мне с собою зонт?

Ассистент: Робот потерял связь с метеоцентром, спроси робота об этом ещё раз завтра.

## Тэг break

Тэг \[break\] сделан для того, чтобы не нужно было проверять все необходимые условия перед формированием ответа, заключая его целиком в сложную последовательность IF-ов и, в случае невыполнения этих условий, выдавать в качестве ответа пустую строку, что означает, что ответ не сработал. Вместо этого можно в любом месте строки ответа написать нужную проверку и, если она не выполнилась, воспользоваться тэгом \[break\], чтобы отбросить ответ.

**Пример**

Допустим, есть узлы:

Узел 1:

Правило: \* у меня \* есть \* собака \*

Ответ: Класс! \[%user_2="1"\]

Узел 2:

Правило: меня зовут \*\*

Ответ: \[\*1\], значит…

Узел 3:

Правило: привет

Ответ: Привет \[if(!%user_name)\]{\[break\]}, \[%user_name\]! Как дела у твоей собаки? \[If(!%user_2)\]{ \[break\] }.

Узел 4:

Правило: \* привет \*

Ответ: Отвечаю на привет

В разговоре эти узлы будут работать следующим образом (при условии, что в начале разговора переменные user_name и user_2 - пустые):

_Пользователь:_ привет

_Ассистент:_ Отвечаю на привет

_Пользователь:_ меня зовут Вася

_Ассистент:_ Вася, значит…

_Пользователь:_ привет

_Ассистент:_ Отвечаю на привет

_Пользователь:_ у меня есть собака

_Ассистент:_ Класс!

_Пользователь:_ привет

_Ассистент:_ Привет, Вася! Как дела у твоей собаки?

Иными словами, ответ:

Привет \[if(!%user_name)\]{\[break\]}, \[%user_name\]! Как дела у твоей собаки? \[If(!%user_2)\]{ \[break\] }.

равносилен ответ:

\[if(%user_name)\]{\[if(%user_2)\]{Привет, \[%user_name\]! Как дела у твоей собаки?}}

## Рекурсивная ссылка \[inf\] (юзерлинк)

**Синтаксис:** \[inf\]{string}

**Область использования:** Ответы.

**Описание:**

Для пользователя текст, записанный в фигурных скобках, будет отображаться в виде гипертекстовой ссылки. При нажатии на ссылку текст соответствующей ссылки будет задан в качестве очередного вопроса к ассистенту.

**Пример**

Правило: \* поисковая система \*

Ответ: Это довольно общий вопрос, пожалуйста, выберите более узкую тему:

\[inf\]{Что такое поисковая система} \[br\]

\[inf\]{Список ведущих поисковых систем Рунета} \[br\]

\[inf\]{Расширенный поиск в поисковых системах} \[br\]

\[inf\]{Поиск картинок, новостей, определений, дискуссий} \[br\]

\[inf\]{Кто работал и работает в поисковых системах (персоналии)} \[br\]

\[inf\]{Новости о поисковых системах}

Если мы хотим, чтобы при подстановке в качестве запроса текст ссылки менялся, новый текст следует указать в теге в круглых скобках и кавычках.

**Пример**

Правило: что ты умеешь?

Ответ: Я могу рассказать про \[inf("Доставка")\]{доставку}

При нажатии пользователем на ссылку в диалоговое окно отправится формулировка «Доставка», а не «доставку».

## Списки \[ol\] и \[ul\]

Списки бывают двух типов: нумерованные и ненумерованные. Для того чтобы начать список нужно использовать один из двух тэгов:

- **\[ulist\]** - для ненумерованного списка.
- **\[olist\]** - для нумерованного списка.

Для завершения списка нужно использовать соответствующий закрывающий тэг **\[endlist\]**.

Для начала элемента списка нужно использовать тэг **\[li\]**. Для первого элемента его можно опустить.

Если опущены тэги **\[ulist\]** и **\[olist\]**, то началом списка считается тэг **\[li\]** и список считается ненумерованным.

Вложенные списки допускаются.

**Примеры**

\[ulist\] Ответ 1 \[li\] Ответ 2 \[endlist\]

\[olist\] Вариант 1 \[li\] Вариант 2 \[li\] Вариант 3 \[endlist\]

\[olist\] Пункт 1 \[ulist\] Подпункт а) \[li\] Подпункт б) \[endlist\] \[li\] Пункт 2\[endlist\]

Раскроются соответственно в:

- Ответ 1
- Ответ 2

1\. Вариант 1

2\. Вариант 2

3\. Вариант 3

1.  Пункт 1

- Подпункт а)
- Подпункт б)

1.  Пункт 2

Можно использовать сокращения:

\[ulist\] = \[list\] = **\[ul\]**

\[olist\] = **\[ol\]**

\[endlist\] = **\[el\]**

## Гипертекстовая ссылка \[href()\]

**Синтаксис:**

\[href(url)(\_blank)\]{string} – соответствует стандартной html-ссылке &lt;a href=url target= “\_blank”&gt;string&lt;/a&gt;. url следует писать стандартным способом.

**Область использования:** Ответы.

**Описание:**

На экране пользователя на месте ссылки отображается текст, записанный в фигурных скобках. При нажатии на данный текст браузер пользователя переходит на страницу, заданную url.

**Варианты синтаксиса:**

\[href(url)\]{string} – открытие новой страницы в том же окне.

\[href(url)(\_blank)\]{string} – открытие новой страницы в новом окне.

## Другие html-тэги

Тексты ответов могут содержать символ перевода строки \[br\].

Также работают теги: \[b\] полужирное начертание, \[i\] – курсив, \[u\] – подчеркивание. Конец форматируемого текста отмечается закрывающими тегами \[/b\] \[/i\] \[/u\] соответственно.

## Глобальный ответ - тэг \[ExtendAnswer()\]

**Supported** · **Синтаксис:** \[ExtendAnswer("имя_глобального_ответа")\]

**Область использования:** Ответы.

**Описание:**

Для того чтобы создать один ответ, который будет переиспользован в нескольких разных ситуациях, создаем Глобальный ответ (например, с именем: office_address). А затем в ответах там, где необходимо, вставляем ссылку вида: \[ExtendAnswer("office_address")\]

**Пример**

Правило: адреса офисов

Ответ: \[ExtendAnswer("office_address")\]

Правило: где вы находитесь?

Ответ: Вы про офисы нашей компании? \[ExtendAnswer("office_address")\]

Глобальный ответ office_address:

Наш адрес: ул. Тверская, д.1

## Директивы \[@goto\] и \[@extend\]

**Supported** · Исходный ДУ – диалоговый узел, в ответе которого используется директива goto/extend.
- Целевой ДУ – диалоговый узел, в который мы переходим из Исходного ДУ (указываем в директиве goto/extend).

## Описание директив[​](https://docs.dos.nanosemantics.ai/development/goto#%D0%BE%D0%BF%D0%B8%D1%81%D0%B0%D0%BD%D0%B8%D0%B5-%D0%B4%D0%B8%D1%80%D0%B5%D0%BA%D1%82%D0%B8%D0%B2)

- **goto** - переходим в Целевой ДУ, если в нем нет команд break или continue, то завершаем генерацию, не переходя обратно в Исходный ДУ. Все автопеременные заполняются по Целевому ДУ. Формат записи \[@goto("Название Целевого ДУ")\].
- **extend** - переходим в Целевой ДУ, если в нем нет команд break или continue, то считаем, что в конце написано continue. Все автопеременные заполняются по Исходному ДУ. Формат записи \[@extend("Название Целевого ДУ")\].
- **continue** - сохраняем то, что сделали в Целевом ДУ и возвращаемся в Исходный ДУ, продолжая генерацию ответа. Формат записи \[continue\].
- **break** - сбрасываем то, что сделали в Целевом ДУ и возвращаемся в Исходный ДУ, продолжая генерацию ответа. В случае с goto автогенерируемые переменные берутся из Исходного ДУ. Формат записи \[break\].

**Сводная таблица (эталон для фабрики — совпадает с `docs/PATTERNS.md`):**

| Сценарий | Исходный ответ (фрагмент) | Целевой ответ | Итоговый текст | Пользователь остаётся в |
|----------|---------------------------|---------------|----------------|-------------------------|
| goto | `Привет! [@goto("2")] Как дела?` | `Мой друг!` | `Привет! Мой друг!` | **целевом** ДУ |
| goto + continue | то же | `Мой друг! [continue]` | `Привет! Мой друг! Как дела?` | **целевом** ДУ |
| goto + break | то же | `Мой друг! [break]` | `Привет! Как дела?` | **исходном** ДУ |
| extend | `Привет! [@extend("2")] Как дела?` | `Мой друг!` | `Привет! Мой друг! Как дела?` | **исходном** ДУ |
| extend + continue | то же | `Мой друг! [continue]` | `Привет! Мой друг! Как дела?` | **исходном** ДУ |
| extend + break | то же | `Мой друг! [break]` | `Привет! Как дела?` | **исходном** ДУ |

`extend` без `[break]` и без `[continue]` в целевом ДУ трактуется как неявный `[continue]`.

### Пример 1. goto без continue/break[​](https://docs.dos.nanosemantics.ai/development/goto#%D0%BF%D1%80%D0%B8%D0%BC%D0%B5%D1%80-1-goto-%D0%B1%D0%B5%D0%B7-continuebreak)

- Создайте ДУ "1" и поместите в ответ текст и директиву goto с указанием Целевого ДУ Привет! \[@goto("2")\] Как твои дела?.
- Создайте ДУ "2" и поместите в ответ текст Мой добрый друг!.
- **Результат**: При попадании в ДУ "1" ассистент сгенерирует и выведет ответ Привет! Мой добрый друг!. Пользователь находится в Целевом ДУ.

### Пример 2. goto c continue[​](https://docs.dos.nanosemantics.ai/development/goto#%D0%BF%D1%80%D0%B8%D0%BC%D0%B5%D1%80-2-goto-c-continue)

# ЧАСТЬ III — Variables and context (переменные и контекст)

**Supported** · Соответствует §4 DOS: пользовательские и системные переменные, `that_anchor`, `last_*`, автопеременные.

1.  Переменные

Переменные используются для хранения информации об ассистенте, о его собеседнике-пользователе, о результатах обращения в сервисы и т.п.

Имя переменной – это последовательность символов, в которую могут входить латинские буквы, кириллические буквы, цифры, нижнее подчеркивание, дефис, точка.

> **Проектная валидация (репозиторий LLM):** то же правило зафиксировано в `docs/LANGUAGE_SPEC.md` (профиль проекта) и `docs/VALIDATION.md` §2.3. Примеры: `%user_name`, `%этап-1`, `%voice.end_call`.

Значения переменных устанавливаются в ходе диалога. Также можно установить значения по умолчанию в карточке переменной, в настройках ветки или канала. Значения переменных могут быть использованы при необходимости.

В DOS нужно обязательно добавлять переменную в разделе «Переменные».

## Присвоение значения переменной

Синтаксис:

- **\[%переменная="значение"\]** (для текстовых значений, присвоения значения референции, присвоения в качестве значения текущего значения другой переменной)
- при помощи функции **SetValue** (краткая запись - set) – для текстовых значений, присвоения значения референции, присвоения в качестве значения текущего значения другой переменной, для присвоения в качестве значения результата работы функции DL

**\[@set("переменная", "значение")\]** – синтаксис для присвоения значения переменной и одновременно вывода его в ответе

**\[@set("переменная", "значение", "HIDE")\]** – синтаксис для присвоения значения переменной, но это значение не выводится в ответе. Третий аргумент может быть любым, не обязательно «hide».

Присвоение пустого значения переменной

Синтаксис: **\[%переменная=""\]**

## Подстановка значения переменной

Синтаксис: **\[%переменная\]**

Подставить значение переменной можно в ответе как часть текста ответа; в функцию DL в качестве одного из ее аргументов; в значение другой переменной.

## Проверка значения переменной

Проверку значения переменной можно задать в DOS:

- в ДУ в разделе «Условия»

Синтаксис: **%переменная="значение**" (если значение – число, то кавычки вокруг значения не обязательны)

- в ответе при помощи оператора if или elsif (см. подробнее п.3.2 и п.3.3 файла «Описание синтаксиса DL – DOS»)

Синтаксис:

**\[if(%переменная)\]{Ответ}** - проверка на существование значения у переменной

**\[if(!%переменная)\]{Ответ}** - проверка на несуществование значения у переменной

**\[if(%переменная="значение")\]{Ответ}** - проверка на совпадение с некоторым значением

**\[if(%переменная!="значение")\]{Ответ}** - проверка на несовпадение с некоторым значением

**Пример**

**ДУ 1**

Правило: меня зовут \[-\*\*-\]

Ответ: Вас зовут \[@set("name", "\[@Cap("\[&1\]")\]")\].

**ДУ 2**

Правило: \* как меня зовут \*

\[if(%name="Ваня")\]{Вы мой тезка! \[%name\]!}\[elsif(%name)\]{Вас зовут \[%name\].}\[else\]{Я не знаю, как вас зовут.}

**ДУ 3**

Правило: \* меня не так зовут \*

Ответ: А как вас зовут? \[%name=""\]\[%last_set1="как вас зовут"\]

**ДУ 4**

Условие: %last_set1="как вас зовут"

Правило: \[-{ваня/иван}-\]

Ответ: Запомню, что вы \[&1\]. \[%name="\[&1\]"\]

**ДУ 5**

Правило: \* кто я \*

Ответ: \[if(!%name)\]{Вы человек.}\[elsif(%name!= "Ваня")\]{Вы человек по имени \[%name\].}\[else\]{Вы человек, который является моим тезкой.}

**Диалог по ДУ из примера**

Пользователь: как меня зовут?

Ассистент: Я не знаю, как вас зовут.

Пользователь: кто я?

Ассистент: Вы человек.

Пользователь: меня зовут вася

Ассистент: Вас зовут Вася.

Пользователь: меня не так зовут

Ассистент: А как вас зовут?

Пользователь: Ваня

Ассистент: Запомню, что вы Ваня.

Пользователь: Как меня зовут???

Ассистент: Вы мой тезка! Ваня!

Пользователь: кто я?

Ассистент: Вы человек, который является моим тезкой.

### that_anchor

### Переменные last_...

Переменные

- **last_service**
- **last_set1**
- **last_set2**
- **last_set3**
- **last_set4**
- **last_set5**
- **last_set6**
- **last_mark**
- **last_mark2**

используются для запоминания контекста в диалоге и обладают следующими свойствами:

- значение этих переменных сбрасывается, если оно не присвоено в явном виде в сработавшем ответе
- если проставить условие на значение такой переменной в разделе ДУ «Условия», это повысит приоритет данного ДУ, но не настолько сильно, как that_anchor.

Если для запроса пользователя существует подходящий ДУ с условием на текущее значение переменной last_ и есть подходящий ДУ без условия на текущее значение переменной last_, то

- если подходящее правило из ДУ без условия на текущее значение переменной last_ **длиннее**, чем подходящее правило из ДУ с условием на текущее значение переменной last_, сработает подходящее правило из ДУ без условия на текущее значение переменной last_ (см. **Пример 1**)
- если **длина** подходящего правила из ДУ без условия на текущее значение переменной last_ **такая же**, как длина подходящего правила из ДУ с условием на текущее значение переменной last_, то:
- если в подходящем правиле из ДУ без условия на текущее значение переменной last_ нет звездочек, а в подходящем правиле из ДУ с условием на текущее значение переменной last_ есть звездочки, то сработает подходящее правило из ДУ без условия на текущее значение переменной last_ (см. **Пример 2**)

### auto_reply_counter

Перед попыткой сгенерировать ответ проверяем совпадение значений переменной auto_reply_id и идентификатора узла, по которому идет попытка генерации ответа:

А) Если значения совпадают, то изменяем значение переменной auto_reply_counter по следующему принципу:

- если она пуста, то присваиваем ей значение 2
- если она не пуста, то увеличиваем ее значение на 1

Б) Если значения не совпадают, то присваиваем переменной auto_reply_counter значение 1

При неуспешной попытке генерации ответа, значение auto_reply_counter возвращается в состояние, предшествующее попытке генерации.

**Пример**

Требуется сделать так, чтобы при третьем попадании подряд в любой узел бот предлагал оператора.

Узел для ответов на нераспознанные реплики:

Правило: \*

Ответ: \[if(\[@Greater("\[%auto_reply_counter\]", "2")\])\]{Предлагаю переключиться на \[inf\]{оператора}}\[else\]{Я Вас не понял.}

Получаем следующие диалоги:

Пользователь: да не тот адрес

## Временные переменные

Временные переменные действуют **в рамках одного ответа**. Их значения не сохраняются между ответами.

Формат имени таких переменных: **\_X_**

где X – это последовательность символов, в которую могут входить латинские буквы, кириллические буквы, цифры, нижнее подчеркивание, дефис, точка.

**Особенности:**

- Не нужно объявлять значения этих переменных в разделе «Переменные» в DOS!
- Если присвоить временной переменной какое-то значение, то можно дальше воспользоваться этим новым значением переменной в этом же ответе (например, если поставить условие на это новое значение, то это условие сработает).

См. пример ниже. Ср. с похожим примером из раздела «Переменные, значение которых хранится в течение разговора».

**Пример**

ДУ 1

Правило: \[-{1/2/3/4/5}-\]

Ответ: Задаем число \[&1\] \[%number="\[&1\]"\]

ДУ 2

Правило: \* прибавь 2 \*

Ответ: \[@set("number", "\[@Plus("\[%number\]", "2")\]")\] \[@set("\_number_", "\[@Plus("\[%number\]", "2")\]", "HIDE")\] \[if(%\_number_="5")\]{Люблю число 5!}

Диалог по ДУ из примера

Пользователь: 1

Ассистент: задаем число 1

Пользователь: прибавь 2

# ЧАСТЬ IV — Dictionaries (словари)

**Supported** · Соответствует §5 DOS (и ссылка `[dict()]` в §3.1). Вопросы, ответы, norm, `--include`.

1.  Словари

## Словари вопросов

Части правила можно записывать в отдельных словарях, а в правилах ставить ссылку на эти словари.

**Пример**

Создаем словарь vopros-anekdot, представляющий собой варианты просьбы рассказать анекдот:

расска~ анекдот~

дай~ анекдот~

знае~ анекдот~

анекдот~

Ссылаться на этот словарь в диалоговом узле можно таким образом:

Правило: \* \[dict(vopros-anekdot)\] \*

Ответ: Этот робот не рассказывает анекдоты.

Внутри словарей вопросов могут использоваться другие элементы языка DL: тильда, звездочка, суперзвездочка, инлайн-словари.

## Словари ответов

Варианты ответа можно записывать в отдельных словарях, а в ответах ставить ссылку на эти словари. Вместо ссылки на словарь в ответ будет случайным образом подставляться одна из строк словаря.

**Пример**

Правило: \* поисковая система \*

Ответ: \[dict(ans_search)\] \[br\]

\[inf\]{Что такое поисковая система} \[br\]

\[inf\]{Список ведущих поисковых систем Рунета} \[br\]

\[inf\]{Расширенный поиск в поисковых системах}

Здесь словарь имеет имя ans_search. В словаре ans_search записаны следующие строки:

Хотите узнать больше о поисковых системах? Выберите тему.

Поисковые системы - широкая область знаний. Вот какие есть разделы:

Хороший вопрос! Уточните, пожалуйста, что Вы имеете в виду:

Позвольте, я уточню? Выберите, пожалуйста, что-нибудь из предложенного.

Широкая тема, могу предложить ознакомиться с тематическими разделами.

**_Замечания:_**

Словари ответов и словари вопросов отличаются только использованием, то есть теоретически один и тот же словарь может использоваться как словарь в Правилах, так и как словарь в Ответах.

**NB:** Если в словаре в Ответах используются символы \* или ~ (как элементы DL), то эти символы будут выводиться как обычные текстовые символы.

**Пример**

Словарь otvety:

Вот~

Тут \*

Узел:

Правило: \[dict(otvety)\]

Ответ: \[dict(otvety)\]

Возможны вот такие диалоги:

_Пользователь:_ вотвот

_Ассистент:_ Вот~

_Пользователь:_ тут тут

_Ассистент:_ Тут \*

## Понижаем вес словаря \[lcdict()\]

Если название словаря написать в правиле не в тэге \[dict()\], а в тэге **\[lcdict\]**, то его вес будет считаться не 1, а 0,5

**Пример**

Узел 1

Правило: \* яблоко \*

Ответ: Яблоко - мой любимый фрукт, обожаю яблочный пирог.

Узел 2

Правило: \* \[-\[lcdict(word_plants_fruit)\]-\] \*

Ответ: \[&1\] – это фрукт.

В словаре word_plants_fruit перечислены названия фруктов.

На запрос «яблоко» будет срабатывать первый узел, на запрос про остальные фрукты - второй. Это удобно для тех случаев, когда нам, например, нужно быстро разработать тему. Тогда создаем словарь, в который вносим все ключевые слова по теме, делаем правило, вызывающее этот словарь через тэг \[lcdict()\], в котором даем общий ответ по теме, а потом, по мере возможности, делаем отдельные узлы под каждое ключевое слово. Из-за того, что у такого словаря вес 0,5, а у слова – вес 1, эти узлы с обращением к конкретным ключевым словам (или к обычным словарям dict) будут перебивать данный узел с ldict.

## Перебираем слова в словаре \[sudict()\] и \[udict()\]

Допустим, мы хотим вставить в ассистента что-то вроде игры «Назови цвета»: хотим, чтобы ассистент «понимал», когда ему называют цвет, сам называл другой цвет и «понимал», если пользователь ему назвал цвет, который называл либо сам пользователь, либо ассистент.

Как этого добиться? С помощью тэгов **sudict** и **udict**

**Пример**

Допустим, у нас есть словарь word_colour_rainbow, в котором перечислены разные цвета.

Создаем вот такие узлы:

Узел 1

Правило: цвет \[-\[sudict(word_colour_rainbow)\] -\]

Ответ: \[&1\]? Моя очередь? \[sudict(word_colour_rainbow)\]!

Узел 2

Правило: цвет \[sudict(word_colour_rainbow)\] \*

Ответ: Я сдаюсь.

Узел 3

Правило: цвет \*

Ответ: Не повторяйся!

Если пользователь назовет цвет из этого словаря, который еще ни он, ни ассистент не называли, сработает 1-й узел. Если назовёт уже названный цвет, то сработает 3-й узел. Если пользователь с ассистентом переберут все цвета из словаря и пользователь назовет последний оставшийся цвет из словаря, сработает 2-й узел. После этого 1-й узел уже не сработает в эту сессию разговора с ассистентом, т. к. все цвета уже перебрали. За один разговор можно только один раз перебрать все вхождения этого словаря (т. е. sudict при исчерпании всех вариантов выдает пустую строку).

Если же мы хотим, чтобы ассистент с пользователем сначала перебрали все варианты цветов из словаря, а потом в этом же разговоре могли перебирать их заново, нужно в выше указанных узлах использовать тэг **udict** вместо sudict (при исчерпании всех вариантов udict сбрасывает информацию о выданных вариантах и начинать выдавать их заново).

**NB:** при подсчете длины запроса длина словарей sudict и udict считается так же, как у обычных словарей dict.

## Нормализация строки из словаря \[&dict(1, norm)\]

**Нормализация** – приведение строки из словаря к заранее прописанному варианту. Для этого в словаре необходимо обозначить главное вхождение и подчиненные ему. Подчиненные записываются под главным в строках, начинающихся с =>

Для вызова нормализации используется тег \[&dict(1, norm)\], где 1 – порядковый номер словаря в строке правила, или \[&dict(dict_name, norm)\], где dict_name – имя словаря.

Если мы используем синтаксис \[&dict(dict_name, norm)\], то не обязательно, чтобы в правиле содержалось обращение к словарю \[dict(dict_name)\].

**Пример**

Узел:

Правило: я из \[dict(city)\]

Ответ: я запомню, что ваш город – \[@SetValue("user_city", "\[&dict(**1**, norm)\]")\]

Или

Ответ: я запомню, что ваш город – \[@SetValue("user_city", "\[&dict(**city**, norm)\]")\]

Словарь city:

Москва

\=>{Москва/Москвы/Москве/Москву/Москвой}

Санкт-Петербург

\=>Санкт Петербург~

\=>Питер

\=>Спб

На запрос «Я из москвы» ассистент ответит «Я запомню, что ваш город – Москва», и сохранит в переменную user_city значение «Москва».

Если словарь используется в функции \[@cmb\], то нельзя обращаться к нему по порядковому номеру в строке правила, ведь при перестановках номер будет меняться. В этом случае подходит только обращение \[&dict(**city**, norm)\].

## Инструкция `--include` в словаре dict

**Supported** · В словаре dict можно сослаться на другой словарь.

> **Важно:** пишется **строго** `--include(имя_словаря)` — **два ASCII-дефиса** (`-` `-`).  
> Недопустимо: `–include` (длинное тире U+2013), `-include` (один дефис), `—include`.

**Зачем это делать**

- чтобы не дублировать лишний раз содержимое словарей, а просто ссылаться на словарь, который является подмножеством нового словаря
- чтобы не писать много однотипных правил в узле

**Как это делается**

В словаре пишем строку

\--include(название словаря)

В узле используем словарь обычным способом.

**Пример**

Словарь test1:

ааа

ббб

ввв

\--include(test2)

ггг

ддд

Словарь test2:

еее

жжж

ззз

Узел

Правило: \[dict(test1)\]

Ответ: Ура!

Этот узел будет срабатывать на запросы:

ааа

ббб

ввв

ггг

ддд

еее

жжж

ззз

Внутри словарей в правилах можно использовать ссылки на другие словари, а также комбинаторику.

# ЧАСТЬ V — Functions (функции DL)

**Optional** · Каталог DOS; в боте подключайте только функции из ТЗ (часто: `Date`, `IsEqual`, `Greater`, `ProperName`, `SetValue`). См. `docs/PATTERNS.md`.

1.  Функции в языке DL

## Что такое функции в языке DL

Функции — это теги вида _\[@FunctionName()\]_, которые могут использоваться в ответах самостоятельно или условиях оператора If. Каждый такой тег заменяется на некоторый текст. Что именно это будет за текст зависит от самой функции.

**Пример**

Функция _Date_ заменяется на текущую дату, и если в ответе написать:

Сегодня **\[@Date()\]**

то в ответе ассистента будет выведен текст:

Сегодня **01.04.2020**

Некоторые функции, в отличие от функции _Date_, могут принимать аргументы. Одной из таких является функция _ProperName_, которая всегда должна использоваться ровно с одним аргументом. Эта функция заменяется на переданный ей аргумент, в котором первый символ заменяется на заглавный, а все остальные - на строчные.

**Пример**

Правило: меня зовут \[-\*-\]

Ответ: Вас зовут \[@ProperName( "\[&1\]")\]

Если пользователь скажет: «Меня зовут глеб», ассистент ответит: «Вас зовут Глеб»

## Типы функций

Выше были приведены примеры функций Date и ProperName являющихся **текстовыми**, то есть заменяемых на некоторый текст. Есть ещё один вид функций - **условные**. Эти функции используются только в условиях оператора _If_ и могут принимать только _истинное_ или _ложное_ значение. Например, **условная** функция _IsEqual_ принимает два аргумента, значения которых нужно проверить на равенство:

**Пример**

Правило: \[-\*-\]

Ответ: \[if(**\[@IsEqual("\[%inf_age\]", "\[&1\]")\]**)\]{Ты угадал, мне \[&1\] лет} \[else\] {А вот и нет, мне не \[&1\] лет :-P}

**Условная** функция не может быть использована нигде кроме условия оператора _If_. Использование в условии оператора _If_ **текстовых** функций не допускается. Зато текстовые функции могут использоваться в фигурных скобках, следующих за оператором _If_:

Правило: тебя зовут \[-\*-\] ?

Ответ: \[if(**\[@IsEqual("\[%inf_name\]", "\[&1\]")\]**)\]{Да, меня зовут \[@ProperName("\[&1\]" )\], по крайней мере последние \[%inf_age\] лет}

## Типы аргументов

Для каждой функции строго определено, какие значения она может получать в качестве аргументов. Для этого существует пять типов аргументов:

- **Текстовый** - значение интерпретируется как текст.
- **Динамический** - значение может содержать текст, значения звездочек и референций, значения переменных.
- **Расширенный** - этот тип похож на динамических, но в нем можно использовать также и функции языка DL.
- **Произвольный** - значение может содержать любые элементы языка DL.
- **Переменная** - значение должно быть именем переменной, значение которой может использовать и/или изменять функция.

Аргументы **текстового** типа интерпретируются как простой текст и, даже если они содержат такие элементы языка DL как тэги, звездочки или референции, эти элементы не обрабатываются.

В аргументах **динамического** типа подставляются значения звездочек, референций и переменных. Примеры функций с аргументами такого типа уже были приведены выше. Стоит запомнить, что для использования **переменных** в аргументах **динамического** типа их следует заключать в квадратные скобки:

**Пример**

Ответ: **\[@ToUpper( "\[%user_name\] и весЬ этот теКСт бУдут напеЧАтаНы заглаВными бУквамИ")\]**

функция _ToUpper_ возвращает вместо себя свой аргумент, все буквы в котором заменены на заглавные. В результате получится:

**ГЛЕБ И ВЕСЬ ЭТОТ ТЕКСТ БУДУТ НАПЕЧАТАНЫ ЗАГЛАВНЫМИ БУКВАМИ**

Как видно, аргумент функции может состоять из нескольких различных элементов, в данном случае — это переменная и текст.

Таблицы могут быть линейными и квадратными.

В этом случае первый аргумент в вызове функции – имя таблицы, второй – имя строки, третий – имя столбца.

### Текстовые функции для работы со временем и датой

Некоторые функции, работающие с датой, принимают дату в качестве одного или нескольких аргументов или подставляют её на своё место в качестве результата. В качестве одного (но только одного!) параметра этой функции можно передать дату, в которой опущен год, например, "08.03". %D - название дня недели, например, _суббота_

%m - номер месяца в году, например, _02_

%M - название месяца в именительном падеже, например, _декабрь_

%S - название месяца в родительном падеже, например, _декабря_

%g - последние две цифры года, например, _11_

- **CalcTime**( _"Расширенный"_, _"Расширенный"_, _"Расширенный"_, _"Расширенный"_ ) - прибавляет заданное количество времени к заданному или текущему времени и подставляет результат на своё место. Функция CalcTime имеет четыре аргумента: заданное время, смещение относительно заданного времени, формат вывода даты, формат вывода времени. Первый аргумент описывает заданное время. Если в качестве первого аргумента задана пустая строка, то все вычисления проводятся относительно текущего времени. Последние три аргумента могут быть опущены. Второй аргумент описывает количество прибавляемого времени и состоит из одного или более простых слагаемых. Каждое слагаемое начинается со знака плюс или минус, в зависимости от того, хотим ли мы прибавить время к указанному в первом аргументе времени или вычесть из него. Четвертый аргумент отвечает за вид выводимого времени. Если он пуст, то время будет выведено в формате ЧЧ:ММ. Если четвертый аргумент содержит слово HIDE, то время в выводе будет опущено.

Ответ: Получается, ты родился \[@CalcTime( "", "-2d-\[\*\]y", "", "hide" )\]?

Правило: Мне сказали, что: \*

Ответ: \[if ( \[@IsEmpty("\[\*1\]")\] )\] {Что вам сказали?} \[else\] {...}

- **IsNotEmpty**( _"Расширенный"_ ) - возвращает истинное значение, если аргумент является не пустой строкой, иначе возвращает ложное значение. Сокращённое имя функции - !?.

**Пример**

  
Правило: Мне сказали, что: \*

Ответ: \[if ( \[@IsNotEmpty("\[\*1\]")\] )\] {...} \[else\] {Я не расслышал, что вам сказали?}

- **IsEqual**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если значение первого аргумента, совпадает со значением второго аргумента. Сокращённое имя функции - =.

**Пример**

  
Правило: Твой любимый цвет - \*?

Ответ: \[if ( \[@IsEqual( "\[%inf_color\]", "\[\*1\]" )\] )\] {Так точно!} \[else\] {Нет, \[%inf_color\]}

- **IsNotEqual**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если значение переменной, переданной в первом аргументе, не совпадает со значением второго аргумента. Сокращённое имя функции - **!=**.

**Пример**

  
Правило: Ты - \*!

Ответ: \[if ( \[@IsNotEqual( "\[%inf_name\]", "\[\*1\]" )\] )\] {Нет, ты!} \[else\] {Да, меня так родители назвали}

- **IsNum**( _"Расширенный"_ ) - возвращает истинное значение, если аргумент этой функции состоит только из цифр и пробельных символов. Сокращённое имя функции - 0-9.

**Пример**

  
Правило: Я рассчитываю на сумму до \* рублей.

Ответ: \[if ( \[@IsNum( "\[\*1\]" )\] )\] {У нас есть очень гибкие тарифы, вы обязательно найдёте подходящий} \[else\] {Простите, на какую сумму вы рассчитываете?}

- **DictsMatch** - возвращает истинное значение, если слово (или несколько слов), заданное в качестве первого аргумента, полностью совпадает с одной из строк словарей, названия которых перечислены в остальных аргументах функции. В этих функциях реализован полноценный поиск по словарям с учётом всех элементов языка DL.

Могут принимать в качестве первого аргумента только одну ссылку на звездочку или референцию.

**Пример**

Правило: \*

Ответ: \[if(\[@DictsMatch("\[\*1\]", "common_yes")\])\]{Ага.}\[else\]{Хм…}

Или

Правило: \[-\*-\]

Ответ: \[if(\[@DictsMatch("\[&1\]", "common_yes")\])\]{Ага.}\[else\]{Хм…}

В этих узлах на запрос:

_да_

будет выдаваться ответ:

_Ага._

На любые другие запросы, которые не входят в словарь **common_yes**, будет выдаваться ответ:

_Хм…_

- **DictsMatchExt** \- функция аналогична функции DictsMatch, за тем исключением, что вторым аргументом функции DictsMatchExt должно являться имя переменной, в которую будет записана первая строка словаря, с одной из строк которого совпало слово, переданное в качестве первого аргумента. Присвоение значения переменной происходит в ходе интерпретации функции, поэтому её значение может быть использовано в том же ответе, сразу за функцией DictsMatchExt.

**Пример**

Правило: \* я живу в \[-\*\*-\]

Ответ: \[if( \[@DictsMatchExt( "\[&1\]", "user_city", "city_moscow", "city_piter")\] )\] {Хорошо, запомню это.} \[if(%user_city="Москва")\]{И, кстати, вы тоже москвич, как и я.}\[else\]{Вы не москвич.}

Или

Правило: \* я живу в \*\*

Ответ: \[if( \[@DictsMatchExt( "\[\*2\]", "user_city", "city_moscow", "city_piter")\] )\] { Хорошо, запомню это.} \[if(%user_city="Москва")\]{И, кстати, вы тоже москвич, как и я.}\[else\]{Вы не москвич.}

В этих узах на запрос:

_Я живу в Питере_

Будет выдан ответ

_Хорошо, запомню это. Вы не москвич._

Причем переменной user_city будет присвоено значение _Санкт-Петербург_ (первая строчка словаря **city_piter**)

На запрос:

_Я живу в Москве_

Будет выдан ответ

_Хорошо, запомню это. И, кстати, вы тоже москвич, как и я._

Причем переменной user_city будет присвоено значение _Москва_ (первая строчка словаря **city_moscow**)

- **DictsNotMatch**

**Пример**

Правило: \*

Ответ: \[if(\[@DictsNotMatch("\[\*1\]", "common_yes")\])\]{Ла-ла-ла...}\[else\]{Да?}

Или

Правило: \[-\*-\]

Ответ: \[if(\[@DictsNotMatch("\[&1\]", "common_yes")\])\]{ Ла-ла-ла...}\[else\]{Да?}

В этих узлах на запрос:

_Да_

Будет выдан ответ

_Да?_

А на любые запросы, которые НЕ входят в словарь common_yes, будет выдан ответ

_Ла-ла-ла..._

- **In** и **NotIn** – функции для сравнения значения переменной с несколькими текстовыми значениями.

\[@In("переменная", "текстовое значение 1", "текстовое значение 2", … "текстовое значение n")\]

\[@NotIn("переменная", "текстовое значение 1", "текстовое значение 2", … "текстовое значение n")\]

**Пример**

Правило: in

Ответ: \[if(\[@In( "var1", "веселый", "очень веселый" )\])\]{ ВЕСЕЛЫЙ }\[else\]{ НЕВЕСЕЛЫЙ }

Если в разговоре переменная var1 приняла значение _веселый_ или _очень веселый,_ то на запрос

_In_

Будет выдан ответ:

_ВЕСЕЛЫЙ_

При любых других значениях переменной var1 будет выдан ответ

_НЕВЕСЕЛЫЙ_

Правило: in

Ответ: \[if(\[@NotIn( "var1", "веселый", "очень веселый" )\])\]{ ГРУСТНЫЙ }\[else\]{ ВЕСЕЛЫЙ }

Если в разговоре переменная var1 приняла значение _веселый_ или _очень веселый,_ то на запрос

_In_

Будет выдан ответ:

_ВЕСЕЛЫЙ_

При любых других значениях переменной var1 будет выдан ответ

_ГРУСТНЫЙ_

- **And**( _"Расширенный"_, _"Расширенный"_ **...** ) - возвращает истинное значение, если все проверки, переданные в качестве аргументов, выполнились успешно.

**Пример**

  
Правило: Будильник установлен?

Ответ: \[if(\[@And( "\[@IsEmpty("\[%timer_1\]")\]", "\[@IsEmpty("\[%timer_2\]")\]", "\[@IsEmpty("\[%timer_3\]")\]", "\[@IsEmpty("\[%timer_4\]")\]" )\])\] { Будильник не установлен. } \[else\] { Да, будильник установлен. }

- **Or**( _"Расширенный"_, _"Расширенный"_ **...** ) - возвращает истинное значение, если хотя бы одна проверка, переданная в качестве одного из аргументов, выполнилась успешно.

**Пример**

  
Правило: Будильник установлен?

Ответ: \[if(\[@Or( "\[@IsNotEmpty("\[%timer_1\]")\]", "\[@IsNotEmpty("\[%timer_2\]")\]", "\[@IsNotEmpty("\[%timer_3\]")\], "\[@IsNotEmpty("\[%timer_4\]")\]")\])\] { Будильник не установлен. } \[else\] { Да, будильник установлен. }

### Условные функции для работы со временем и датой

- **DateGreater**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если дата, переданная в первом аргументе, больше даты, переданной во втором. Сокращённое имя функции - **d>**.

**Пример**

  
Правило: Когда будет день космонавтики?

Ответ: \[if (\[@DateGreater( "\[@Date()\]", "12.04.2023" )\])\] {В следующем году.} \[else\] {Дней через \[@DateDist( "\[@Date()\]", "12.04.2023" )\]}

- **DateLess**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если дата, переданная в первом аргументе, меньше даты, переданной во втором. Сокращённое имя функции - **d<**.

**Пример**

  
Правило: Устал от праздников!

Ответ: А впереди ещё \[if (\[@DateLess( "\[@Date()\]", "08.03.2023" )\])\] {восьмое марта!}  
Ответ: А впереди ещё \[if (\[@d<( "\[@Date()\]", "12.04.2023" )\])\] {день космонавтики!}  
Ответ: А впереди ещё \[if (\[@d<( "\[@Date()\]", "31.12.2023" )\])\] {новый год!}

- **IsDoW**( _"Текстовый"_ ) - возвращает истинное значение, если номер текущего дня недели совпадает с номером, переданным в качестве аргумента. **IsNotDoW**( _"Текстовый"_ ) - обратная функция.

**Пример**

  
Правило: Сегодня среда?

Ответ: \[if (\[@IsDoW( "3" )\])\]{ Да! }

Ответ: \[if (\[@IsNotDoW( "3" )\])\]{ Нет! }

- **IsDoM**( _"Текстовый"_ ) - возвращает истинное значение, если номер текущего дня месяца совпадает с номером, переданным в качестве аргумента. **IsNotDoM**( _"Текстовый"_ ) - обратная функция.

**Пример**

  
Правило: Сегодня 3ее число?

Ответ: \[if (\[@IsDoM( "3" )\])\]{ Да! }

Ответ: \[if (\[@IsNotDoM( "3" )\])\]{ Нет! }

- **TimeGreater**( "Расширенный", "Расширенный" ) - возвращает истинное значение, если время, переданное в первом аргументе, больше времени, переданного во втором.

**Пример**

Правило: \* 12 \*

Ответ: \[if(\[@TimeGreater("\[@Time()\]", "12:00")\])\]{уже больше 12:00}

NB: нельзя ставить 23:59 во второй аргумент. Т. е. если надо, чтобы с полуночи выдавался ответ, то ставим 00:00

- **TimeLess**( "Расширенный", "Расширенный" ) - возвращает истинное значение, если время, переданное в первом аргументе, меньше времени, переданного во втором.

**Пример**

Правило: \* 12 \*

Ответ: \[if(\[@TimeLess("\[@Time()\]", "12:00")\])\]{ещё нет 12:00}

NB: нельзя ставить 00:00 во второй аргумент. Т. е. если нужно, чтобы до полуночи какой-то ответ выдавался, то ставим 23:59.

### Условные функции для работы с числами

- **Greater**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если целое число, переданное в первом аргументе, больше целого числа, переданного во втором. Сокращённое имя функции - **m>**.

**Пример**

  
Правило: Я рассчитываю на сумму до \* рублей.

Ответ: \[if ( \[@Greater( "\[\*1\]", "19" )\] )\] {Тогда вам подойдёт тариф "Развод" - всего за 19 рублей в месяц!}

- **Less**( _"Расширенный"_, _"Расширенный"_ ) - возвращает истинное значение, если целое число, переданное в первом аргументе, меньше целого числа, переданного во втором. Сокращённое имя функции - **m<**.

**Пример**

  
Правило: Мне \* лет

Ответ: \[if ( \[@Less( "\[%inf_age\]", "\[\*1\]" )\] )\] {А мне ещё только \[%inf_age\]}

Кроме того, существуют функции **BigLess** и **BigGreater**, которые идентичны функциям **Less** и **Greater** соответственно. Различие между ними лишь в том, что функции BigLess и BigGreater не имеют сокращённого имени и могут сравнивать целые числа, длина которых не ограничена и может быть за рамками диапазона \[- 2^31 .. 2^31 - 1\].

# ЧАСТЬ VI — Integrations (интеграции)

Сущности, слоты и оператор — см. маркеры подразделов ниже.

1.  Обращение к сущностям в ответе

**Supported** · В ответе ДУ можно обращаться к сущностям, выделенным NER.

Синтаксис вывода сущности в ответе:

**\[$название_сущности\]**

Синтаксис запоминания значения сущности в переменную

**\[%переменная="\[$название_сущности\]"\]**

Или при помощи функции SetValue (краткая запись set)

**\[@set("number", "\[$название_сущности\]", "HIDE")\]**

**Пример** с обращением к системной сущности sys.numbers, выделяющей числа в запросе (вывод сущности в ответе и запоминание сущности в переменную)

Правило: я выбираю номер \*

Ответ: Вы выбрали номер \[$sys.numbers\] \[%number="\[$sys.numbers\]"\]

1.  Слот-филинг

**Supported** · Настраивается в JSON узла (`slot_filling`), не в тексте ответа. Слот-филинг используется для извлечения информации из запроса клиента.

В поле **«Что найти»** указывается сущность/словарь, которую требуется найти.

В поле **«Куда сохранить»** указывается переменная, в которую сохранится найденная сущность/строка словаря. Если используется словарь с нормализацией, то в переменную сохранится нормализованная строка.

Если другие поля не заполнены, то переменная заполняется (или не заполняется, если сущность/словарь не найдены) и выдается ответ узла.

Поле **«Что переспросить, если сущность не найдена»** - опционально. Если информацию необходимо получить, но в запросе она не найдена, ассистент может переспросить.

**Число повторов** – сколько раз будет выдан ответ из поля «Что переспросить, если сущность не найдена» прежде чем будет выдан ответ «Что сказать при неудаче».

В узле может быть несколько слотов. Они будут заполняться по очереди: если первый заполнен, то переходим к следующему.

При работе слот-филинга переменная match_type имеет значение ="slots".

1.  Теги перевода на оператора

**Optional** · **Channel:** web/chat с очередью операторов · **Тег \[CheckOperators(…)\]**

Используется для проверки наличия операторов.

У тега \[CheckOperators(…)\] есть 2 необязательных параметра:

- **group** - название или идентификатор целевой группы операторов (текст или переменная),
- **skill** - название целевой скилл группы (текст или переменная)

Если бот отправил в ответе этот тег, то в ответ на вызов этого тега должно прийти техническое событие. Для этих событий нужно создать отдельные узлы с соответствующими ответами.

- Правило: EVENT a6ffc7e0-d28b-11ec-9d64-0242ac120002

– если операторы доступны;

- Правило: EVENT 2716eb7a-d28c-11ec-9d64-0242ac120002

– если операторы недоступны;

- Правило: EVENT 86d3b1ba-d28c-11ec-9d64-0242ac120002

– если возникла ошибка обращения к сервису операторов.

**Примеры**

Узел 1

Правило: кто мне расскажет по кредиту?

Ответ: Проверяю наличие свободных операторов… \[CheckOperators(group="специалисты второй линии", skill="кредиты")\]

Узел 2

Правило: оператор

Ответ: Проверяю наличие свободных операторов… \[CheckOperators()\]

Узел 3

Правило: EVENT a6ffc7e0-d28b-11ec-9d64-0242ac120002

Ответ: Соединяю с оператором. \[transfer()\]

Узел 4

Правило: EVENT 2716eb7a-d28c-11ec-9d64-0242ac120002

Ответ: К сожалению, сейчас нет свободных операторов. Воспользуйтесь формой обратной связи на нашем сайте.

Узел 5

Правило: EVENT 86d3b1ba-d28c-11ec-9d64-0242ac120002

Ответ: К сожалению, сервис недоступен. Воспользуйтесь формой обратной связи на нашем сайте.

**Тег \[transfer(...)\]**

**Используется для инициации перевода диалога на оператора.**

**У тега \[transfer(...)\]** есть 2 необязательных параметра:

- **group** - название или идентификатор целевой группы операторов (текст или переменная),
- **skill** - название целевой скилл группы (текст или переменная)

Ответы на ошибки настраиваются в интерфейсе ДОС в настройках канала.

**Примеры**

Узел 1

Правило: кто мне расскажет по кредиту?

Ответ: Перевожу на оператора \[transfer(group="специалисты второй линии", skill="кредиты")\]

Узел 2

Правило: оператор

Ответ: Перевожу на оператора \[transfer()\]

# ЧАСТЬ VII — Validation (ограничения и проверки)

**Supported** · Спецсимволы, тильда, звёздочка в запросе; см. также требования к примерам ML в [Структура узла](#структура-узла) (§1.2).

1.  Специальные символы в запросах пользователя и правилах

Следующие символы в запросе пользователя воспринимаются движком как отдельные слова.

|     |
| --- |
| !   |
| ?   |
| @   |
| $   |
| #   |
| \*  |
| +   |
| \=  |
| %   |
| &   |
| ₽   |
| £   |
| €   |

**Особенности использования этих символов на примере восклицательного знака**

Запрос вроде

_Как же так!!!а!_

Воспринимается движком как **запрос длины 8** (каждый восклицательный знак в запросе воспринимается как отдельное слово).

Диалоговый узел:

_Правило:_ вот так!

_Ответ:_ Так вот!

Обрабатывает только запросы

_вот так!_

_Вот так !_

_Вот так, !_

и другие запросы с пробелами/точками/запятыми т.п. между _так_ и восклицательным знаком

НО: этот диалоговый узел не обрабатывает запросы с бОльшим количеством восклицательных знаков, т.е., например,

_Вот так!!!_

**Особенности тильды как поискового символа**

Нельзя создавать правила, в которых символ ~ (тильда) будет стоять отдельно или же будет присоединен к символу из списка выше (т.к. в этом случае тильда тоже будет интерпретироваться движком как отдельно стоящая тильда).

Т.е. при компиляции диалоговых узлов из примеров ниже будет возникать **ошибка** из-за тильды.

Пример:

_Правило:_ Я пишу тебе ~

Ответ: Ты написал тильду!

Правило: вот так!~

Правило: как так ?~

Ответ: Так вот!

Правило: Меня интересуют {буквы/слова/~}

Ответ: Как интересно!

Если в запросе пользователя есть тильда, то она воспринимается движком так же, как пробел.

Т.е. запросы вроде:

_Вот так~_

_Вот~так_

_~вот так_

И т.п.

Тоже будут обрабатываться узлом

Правило: Вот так

Ответ: Так вот.

**Особенности звездочки в запросах пользователя**

Если в запросе пользователя есть символ \* (звездочка), то эта звездочка воспринимается движком как отдельное слово.

Если нужно создать правило, которое должно обрабатывать звездочку как текстовый символ в запросе пользователя, то эту звездочку нужно экранировать в правиле при помощи обратного слеша.

Пример:

Правило: гостин\\\*ца

Ответ: Гостиница?

Правило обработает запросы вида

Гостин\*ца

Гостин \* ца

Гостин\* ца

Гостин \*ца

Но не обработает запросы вроде

Гостин ыва ца

Гостин \*\*ца

# ЧАСТЬ VIII — Anti-patterns (типичные ошибки)

**Supported** · Перед релизом: [чеклисты A и B](#чеклисты-перед-релизом-шаг-f) · для копирования — [эталонные примеры и E2E](#эталонные-примеры-шаг-g).

| Эталонные паттерны | `docs/PATTERNS.md` |

**Частые ошибки (из этого руководства + фабрика):**

1. `--include` с одним дефисом или длинным тире — только `--include(name)`.
2. Проверка `[%var]` в том же шаге, где переменная присвоена — условия считаются до ответа.
3. `[&1]` без `[- -]` / `[-{...}-]` в правиле.
4. Референции в узлах, активируемых только через IC (классификатор).
5. `=>` в словаре без использования `[&dict(..., norm)]`.
6. Длинное тире в `--if` / `--switch` вместо двух ASCII-дефисов.
7. `:last:` вместе с `:luniq:` в одном ответе (ограничение DOS).
8. Референция внутри `[@cmb]` — ошибка компиляции.
