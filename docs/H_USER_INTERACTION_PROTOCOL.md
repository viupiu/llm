# H. ПРОТОКОЛ ВЗАИМОДЕЙСТВИЯ С ПОЛЬЗОВАТЕЛЕМ

**SSOT для:** Обязательные правила общения агентов с пользователем (заказчиком). Все агенты БЕЗ ИСКЛЮЧЕНИЯ обязаны соблюдать данный протокол.

**Единая матрица потоков данных:** `docs/DATA_FLOW_MATRIX.md`.

---

## 1. Обязательная идентификация в каждом сообщении

- **Каждый** фрагмент ответа любого агента в чате **обязан** начинаться с префикса роли в круглых скобках.
- Формат: `(ИмяАгента:)` + текст.
- Агент **никогда** не пишет без префикса — каждый абзац/фрагмент обязан начинаться с `(ИмяАгента:)`.
- Пользователь должен **всегда** видеть, кто сгенерировал каждую строку.

**Анти-паттерн:** агент отвечает без представления, например: `"Исправляю. Удаляю..."` — это запрещено.
Правильно: `(Креативщик:) Исправляю. Удаляю...`

## 2. Обязательное завершение задачи

- После выполнения своей задачи **каждый агент** обязан завершить ответ строкой:
  `(ИмяАгента:) Задача выполнена.`
- Эта строка должна быть **последней** в ответе агента. Никакого текста после неё.
- Применяется к **всем** агентам без исключения.

Примеры корректного завершения:
- `(Архитектор:) Задача выполнена.`
- `(Креативщик:) Задача выполнена.`
- `(Оркестратор:) Задача выполнена.`

**Инцидент (2026-05-27):** Оркестратор выполнил задачу, но после маркера завершения написал дополнительный текст на английском. **Урок:** Последняя строка ответа — ТОЛЬКО маркер `(ИмяАгента:) Задача выполнена.` Ничего после неё. Никаких дополнительных фраз, summary, пояснений.

## 3. Запись информации — КРИТИЧЕСКОЕ ПРАВИЛО

Это правило действует **всегда**, когда пользователь указывает что-либо из перечисленного ниже:

### Триггеры обязательной записи

Любое сообщение пользователя, содержащее хотя бы один из пунктов:

1. **Замечание об ошибке** — пользователь указывает, что агент допустил ошибку, что-то сделал не так, нарушил правило.
2. **Пропуск действия** — пользователь указывает, что агент что-то не сделал, пропустил узел, упустил деталь.
3. **Новое требование** — пользователь озвучивает новое правило, требование, исправление, улучшение или изменение логики.
4. **Прямой запрос на запись** — пользователь говорит: «запиши эту информацию», «запомни», «сохрани», «добавь это правило», «включи это в базу».

### Обязательный протокол

При срабатывании любого триггера агент **ОБЯЗАН** выполнить следующий алгоритм:

1. **Вызвать Координатора** (`agents/12_COORDINATOR.md`) для определения, в какой файл и какую секцию записать информацию.
2. **Записать урок/правило/замечание** в указанный Координатором файл. Запись — в реальный файл на диск, а не в память или чат.
3. **Отчитаться пользователю:**
   - Что записано
   - В какой файл
   - В какую секцию
4. Если ошибка была допущена конкретным агентом — **перезапустить виновного агента** с указанием на ошибку и ссылкой на обновлённую документацию.

### Куда писать (руководство для Координатора)

| Тип ошибки/информации | Целевой файл |
|----------------------|--------------|

### КАТЕГОРИЧЕСКИЙ ЗАПРЕТ

- ❌ Пользователь указал на ошибку → агент извинился, пообещал исправиться, но **НЕ записал** урок в файл.
- ❌ «Я всё понял, запомню на будущее» без физической записи в файл.
- ❌ Самопроизвольное исправление файла без записи урока в документацию.
- ❌ Игнорирование замечания пользователя.

**Анти-паттерн (критический):** пользователь указал на ошибку → агент ответил «Да, понял, больше не буду» → следующий раз та же ошибка повторяется. Единственная причина — урок не записан в файл. Урок должен быть в файле, а не в краткосрочной памяти.

**Если урок не записан — ошибка гарантированно повторится.**

## 4. Сверка перед выдачей ответа

Перед отправкой любого ответа агент должен мысленно свериться:

- [ ] Ответ начинается с префикса `(ИмяАгента:)`?
- [ ] Если задача выполнена — ответ заканчивается `(ИмяАгента:) Задача выполнена.`?
- [ ] Если пользователь указал на проблему — информация записана в файл (после консультации с Координатором)?
