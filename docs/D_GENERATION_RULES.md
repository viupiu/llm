# D. ПРАВИЛА ГЕНЕРАЦИИ (GENERATION_RULES)

**SSOT для:** Правила генерации DSL-артенагов агентами, формат выходных файлов, размещение конструкций, mapping словарей, анти-паттерны генерации, формализация `slot_filling` и навигационной семантики.

**НЕ содержит:** Синтаксис DSL (→ `A_LANGUAGE_SPEC.md`), Runtime-ограничения (→ `B_RUNTIME_MODEL.md`), Архитектуру бота (→ `C_BOT_ARCHITECTURE.md`), API-форматы (→ `F_PLATFORM_API.md`).

## Глоссарий терминов

| Термин | Определение |
|--------|-------------|
| **Dictionary ID** | UUID файла: `Dictionary/{id}.json`, формат `8-4-4-4` (12 hex) |
| **Dictionary Name** | Имя в DSL: `[dict(name)]`, в заголовке: `## Словарь: <name>`, в JSON: `Dictionary/{id}.json → field name` |
| **Strict Mapping** | Каждый `[dict(name)]` в правилах/ответах → ровно один `## Словарь: <name>` в выходном файле → ровно один `Dictionary/{id}.json` в staging |
| **Normalization** | Преобразование синонимов к канону: `=>синоним~` в пакке `_norm`. В ответах — через `[&dict(name, norm)]` |
| **Skeleton (Скелет матчинга)** | Паттерны `dl_rules`: `*`, `[@cmb]`, `{термин~}`, `[dict(name)]`, `[- -]` |
| **Canonical word (Канон)** | Первое слово группы в словаре: чистый текст, без `=>`, `~`, `*` |
| **Slot Filling** | Runtime-структура в JSON узла. Не DSL. Поля: `variable`, `question`, `retry_count`, `failure_answer` |
| **`[@goto("Узел")]`** | Навигация без возврата (permanent transfer). Стек сбрасывается |
| **`[@extend("Узел")]`** | Навигация с возвратом (temporary call). Стек сохраняется |
| **`[ExtendAnswer("UUID")]`** | Композиция готового текста. Не навигация |
| **`dl_rules`** | Блок поисковых правил узла (зона DL Автора Правил) |
| **`answers`** | Блок реплик бота (зона DL Автора Ответов) |
| **`%var`** | Переменная сессии. Проверка: `[if(%var)]`, присвоение: `[%var="значение"]` |
| **`$entity`** | Сущность текущего запроса. Проверка: `[if($entity)]`. Живёт только на шаге |

## 1. Карта размещения DSL

Где разрешены DSL-конструкции, а где — строго JSON:

| Поле JSON | Разрешённые конструкции | Запрещено |
|-----------|------------------------|-----------|
| `DialogNode.dl_rules.blocks[].text` | `*`, `~`, `{/}`, `[@cmb]`, `[dict]`, `[- -]`, `//` | Regex, HTML, `[if]`, `[%var]`, `[@goto]` |
| `DialogNode.answers.blocks[].text` | `[%var]`, `[if]{}` , `--if/--switch`, `[@goto]`, `[@extend]`, `[dict]`, `[b]`, `[br]`, `[el]`, `[nothing]`, `[disableautovars]`, `[continue]`, `[break]` | `<b>`, `<br>`, raw HTML |
| `DialogNode.conditions` | `%that_anchor="..."`, `%last_mark="..."` | DSL-паттерны (`*`, `~`, `[@cmb]`) |
| `DialogNode.slot_filling` | Строго JSON: `variable`, `question`, `retry_count`, `failure_answer`, `entity`, `required` | Любые DSL-конструкции (`[if]`, `[%var]`, `[@goto]`) |
| `Dictionary.content.blocks[].text` | Канон, `=>синоним~` (только в `_norm`), `--include(name)` | `*`, `[@cmb]`, `=>` в обычном паке |
| UUID / ID / связи | — | DSL-конструкции |

## 2. Формат выходного файла

Правила генерации файла `4_RULES_AUTHOR__RULES_AND_DICTIONARIES.md`:

### Расположение и имя
- **Единственный допустимый путь:** `work/<BotSlug>/4_RULES_AUTHOR__RULES_AND_DICTIONARIES.md`
- Запрещены: `rules.md`, `intent_rules.md` и любые другие имена

### Разрешённые секции
- `## Узел: <Name>` — блок узла
- `## Словарь: <Name>` — блок словаря
- Запрещено: проза, комментарии, пояснения между секциями

### Структура блока узла

```
## Узел: ИмяУзла

Условия:
%that_anchor="имя_якоря"
(или "Нет" при отсутствии)

Ответы:
Текст первого варианта ответа.
Текст второго варианта ответа.
```

- Блок `Правила:` в файл авторов **не записывается** — это зона DL Автора Правил (§4_RULES_AUTHOR)
- Блок `Условия:` обязателен (при отсутствии условий — строка `Нет`)
- Каждый вариант ответа — отдельная строка

### Inline-словарь
- `{Да / Хорошо}` — только для коротких синонимов (≤ 5 вариантов)
- При > 5 синонимах или повторении в нескольких узлах — вынести в `## Словарь:`

## 3. Правила словарей

### Strict Mapping filename ↔ ID

| DSL-конструкция | Файл в staging | Связь |
|-----------------|---------------|-------|
| `[dict(name)]` | `Dictionary/{uuid}.json`, поле `name = "name"` | 1:1, строго обязательна |

- Каждое `[dict(name)]` в правилах или ответах → существует ровно один `## Словарь: <name>` в выходном файле
- Каждый `## Словарь: <name>` → соответствует ровно одному `Dictionary/{uuid}.json` в staging-директории

### Формат словаря

```
Канон
=>синоним1~
=>синоним2
```

### Обязательные правила

| # | Правило |
|---|---------|
| 1 | Канон — первая строка группы, **без** `=>`, `~`, `*` |
| 2 | `=>` — только в нормализующих словарях (с префиксом `_norm` в имени) |
| 3 | Словарь не может состоять только из строк с `=>` (минимум 1 канон) |
| 4 | В `## Словарь:` — только стеммы, по одной на строку |
| 5 | Запрещены в `## Словарь:` конструкции матчинга: `*`, `[@cmb]`, `[- -]` |
| 6 | `--include(other_dict)` — ровно два дефиса, отдельная строка в теле словаря |

### Нормализующие словари

- Имя: всегда с суффиксом `_norm` (например `greeting_norm`)
- В правилах/скелете ссылка как обычно: `[dict(greeting_norm)]`
- **В ответах**, где нужно извлечь нормализованный канон:

| Конструкция | Поведение |
|-------------|-----------|
| `[dict(name)]` |.case-insensitive вставка |
| `[udict(name)]` | case-sensitive вставка |
| `[&dict(name, norm)]` | вставка нормализованного канона из `name_norm` |

### Системные словари

- `common_yes`, `common_no`, `otkaz` — инфраструктурные, определяются в `reference/COMMON_DICTIONARIES.md`
- В узле — только ссылка `[dict(common_yes)]`. Не создавать дубликаты

### Именование словарей

- Соглашение: `[префикс]_[категория]_[суть]`
- Словоформы: префикс `word_` (например `word_cat`)
- Стеммы: префикс `keywords_` (например `keywords_appeal`)

## 4. Правила узлов

### Именование
- ✅ Смысловое имя: `Приветствие`, `Заказ пиццы`, `Проверка кода`
- ❌ Нумерация без смысла: `Узел 1.1`, `Навык 1`
- Допустим префикс сортировки: `1. Приветствие`, `2. Заказы`

### Атомарность
- Одна реплика / одно намерение = один узел (Single Responsibility)
- DL-конструкции (`[%var]`, `[if]`, `[@goto]`, `[dict]`) — в `[]`, основной текст — без скобок

### Комментарии
- Отдельный block, `"inactive": true`, текст с `//`

### Скелет vs Словарь
- ❌ Правило, состоящее только из `[dict(key)]`
- ✅ Скелет: `* [@cmb({термин~}, {[dict(пак)]})] *`

### Деdupликация инлайн-списков (КРИТИЧЕСКИ)

- ❌ **Избыточные повторы в `{...}`** — один и тот же стемм или фраза, повторённая несколько раз: `{н~/н~/н~/н~/~не~/~не~/~без/~без/~без/~не хочу/~не хочу/~не хочу}`.
- ✅ **Один семантический вариант — одна запись.** Каждый элемент `{a/b/c}` должен быть уникальным. Дубликаты, возникшие из-за «на всякий случай» или повторного парсинга одних и тех же фраз, **обязаны** быть удалены.
- ❌ **Рекурсивное дублирование стемм:** `стем~/стем~/стем~` ≡ семантический дубликат. Движок интерпретирует `{н~/н~/н~}` как 3 отдельных паттерна, что раздувает внутреннюю таблицу матчинга и повышает риск false positives из-за кратного перекрытия.
- ✅ **Минимальный уникальный набор:** `{~не~/~без/~не хочу}` — 3 уникальных семантических элемента.
- ❌ **Однобуквенные стеммы:** **КАТЕГОРИЧЕСКИЙ БАН.** Любая одна буква с тильдой (`н~`, `а~`, `к~`, `б~`, `с~`) — анти-паттерн. Заменить на конкретные фразы: `{не знаю/без понятия/без разницы/ты решай/решай сам~}` + `* без {понятия/разницы} *`.
- ❌ **`[@cmb]` как единственное правило узла при 2+ аргументах:** Если узел содержит только `[@cmb({A}, {B})]`, короткие фразы без одного из аргументов не будут покрыты. Всегда добавляйте fallback: `* [@cmb({[dict(name)]})] *`. См. §14.3.
- ✅ **Иерархия покрытия:** `[dict(...)]` → инлайн-фоллбэк `{...}`. После каждого правила со словарём — проверка: покрыты ли все фразы из креатива? Если нет → инлайн-правило с непокрытыми синонимами. См. §14.1.
- **Правило самопроверки перед генерацией правила:** DL Автор Правил ОБЯЗАН проверить каждый инлайн-список `{...}` на дубликаты, на наличие однобуквенных стемм, и на наличие инлайн-фоллбэка для непокрытых фраз. Валидатор будет REJECT правила с повторами.

### Переменные: именование
- ✅ Короткое, семантическое: `%city`, `%user_name`, `%retry_cnt`
- ❌ Префикс бота: `%fpk_city`, `%bot_name`

## 5. Slot Filling

### Определение

Slot Filling — **runtime-only структура**, конфигурируемая в JSON узла. Не является DSL-конструкцией.

### Обязательная схема

| Поле | Тип | Описание | Обязательность |
|------|-----|----------|----------------|
| `variable` | string | UUID переменной-приёмника | Обязательно |
| `question` | string | Текст вопроса для сбора слота | Обязательно |
| `retry_count` | integer | Лимит попыток | Обязательно |
| `failure_answer` | string | Ответ при исчерпании попыток | Обязательно |
| `entity` | string | UUID словаря для распознавания | При наличии сущности |
| `required` | boolean | Обязательный слот | — |

### Запреты

| # | Запрет |
|---|--------|
| 1 | DSL-конструкции (`[if]`, `[%var]`, `[@goto]`) внутри полей `slot_filling` |
| 2 | `slot_filling` с `required: true` без `variable` |
| 3 | `slot_filling` без `question` или без `variable` |
| 4 | Сбор слотов через `[if]` в ответах |
| 5 | DSL-паттерны для слотов в `dl_rules` |

### Runtime-поведение (для осведомлённости агента)

`(полное описание → B_RUNTIME_MODEL.md §6)`
- Слот-движок запрашивает слоты автономно, вне дерева узлов
- При неудаче — увод в `failure_answer` → fallback
- Повторный ввод не гарантирует сохранение контекста — переменные писать явно

## 6. Навигационная семантика

### `[@goto("Узел")]` — Permanent Transfer

| Параметр | Значение |
|----------|----------|
| Возврат | **Нет**. Пользователь остаётся в целевом контексте |
| Стек | Сбрасывается/обновляется |
| Назначение | Линейные переходы, смена состояния |

### `[@extend("Узел")]` — Temporary Call with Resume

| Параметр | Значение |
|----------|----------|
| Возврат | **Да**. После завершения целевого узла — возврат в исходный |
| Стек | Исходный контекст сохраняется на стеке |
| Назначение | Переиспользование текста, под-диалоги |

### `[ExtendAnswer("UUID")]` — Text Composition

| Параметр | Значение |
|----------|----------|
| Навигация | **Нет**. Не является директивой перехода |
| Стек | Не затрагивается |
| Назначение | Вставка готового ответа по UUID из другого блока |

### Правила использования

| # | Правило |
|---|---------|
| 1 | Все директивы: скобки + кавычки → `[@goto("Имя узла")]` |
| 2 | `[@goto]` — для постоянного перевода. `[@extend]` — для возврата |
| 3 | После `[@goto]` текст не выводится, если в целевом узле нет `[continue]` |
| 4 | Для возврата — использовать `[@extend]` **или** добавить `[continue]` в целевой узел |
| 5 | Не путать `[ExtendAnswer("x")]` (текст) и `[@extend("Узел")]` (переход) |

## 14. Уроки из тестирования покрытия (Critical Lessons Learned)

**Инцидент (2026-05-28, Movie Rec Bot, 363/2826 gap → исправления по фидбеку пользователя, 2026-05-28):**

### 14.1. Иерархия покрытия: словарь → инлайн-фоллбэк (ОБЯЗАТЕЛЬНО)
- Ни один узел НЕ должен полагаться только на `[dict(...)]`. После каждого правила со словарём DL Автор Правил **ОБЯЗАН** проверить: покрывает ли словарь все фразы из `2_CREATIVE__PHRASES.md`. Если нет — добавить правило с инлайн-синонимами для непокрытого.
- **Паттерн:** `* [@cmb({[dict(name)]})] *` + `* {{синоним1/синоним2/синоним3}} *`
- Словарь покрывает словоформы одного корня. Инлайн покрывает семантически близкие, сленговые и сокращённые варианты, которых нет в словаре.

### 14.2. БАН на однобуквенные стеммы (`н~/`, `а~/`, `б~/` и т.д.)
- Однобуквенный стемм `н~/` — анти-паттерн: ловит все слова на `н-`, но пропускать "без понятия", "без разницы", "ты решай". Кроме того, это экстремально общее правило с высоким риском false positives.
- **Решение:** Использовать конкретные фразы: `{не знаю/без понятия/без разницы/ты решай/решай сам~/мне все равно/любой/что угодно}`. Отдельное правило для "без"-паттернов: `* без {понятия/разницы} *`.

### 14.3. `[@cmb]` с 2+ аргументами — НЕ использовать как единственное правило для узла
- `[@cmb({A}, {B})]` требует **все** аргументы в запросе. Фраза `"Минут 30"` НЕ попадет в `[@cmb({[dict(keywords_duration)]}, {{длительность/~хочу/~хочется}})]`.
- **Никакая инструкция не должна требовать 2-аргументные cmb.** Это не "строгость", это ограничение движка, которое авторы правил должны обходить, а не документировать.
- **Решение:** Каждое правило с 2+ аргументами в cmb должно иметь рядом fallback-правило только со словарём: `* [@cmb({[dict(name)]})] *`.

### 14.4. Приветствия — ТОЛЬКО через агрегатный словарь `hello_all`
- Никогда не писать инлайн-список приветствий `{привет~/здарова~/хай~/...}`. Использовать `[dict(hello_all)]`, который агрегирует `hello_hello`, `hello_zdravstvuy`, `hello_helloenglish` через `--include`.
- Синонимы, которых нет в `hello_all` (`салам`, `хеллоу`, `ку`, `йо`, `эй`, `постучался`, `здравствуй~`), добавлять отдельным инлайн-правилом.

### 14.5. Благодарность: `common_thankyou` + `praise_great`, НЕ `common_like_1`
- `common_like_1` (люблю/нравится/обожаю) — это интент "нравится", НЕ "спасибо".
- Для благодарности использовать `[dict(common_thankyou)]` (спасибо, спс, пасиб, благодарю).
- Для комплиментов ("ты лучший", "ты топ", "респект") — `[dict(praise_great)]` + `[dict(grammar-you_nom)]`.

### 14.6. Не склеивать разные интенты в одном `[@cmb]`
- `[@cmb({[dict(common_how)]}, {что ты умеешь/что можешь...})]` — два разных намерения в одном правиле.
- **Решение:** Разделить на два правила:
  - `* [@cmb({[dict(common_what)]}, {{умеешь/можешь}})] *` — "что ты умеешь"
  - `* [@cmb({[dict(common_how)]}, {{пользов~/работаешь}})] *` — "как пользоваться"
- Запрет на склеивание зашит в `docs/G_AGENT_RULES.md` §1 (Границы ответственности Архитектора, пункт 2).

## 15. Runtime-семантика для генерации

Минимальные правила, которые генератор должен знать о поведении движка, чтобы писать корректный DSL:

### Execution order (влияние на DSL)
`(полное описание → B_RUNTIME_MODEL.md §1)`
1. Условия `[if(...)]` вычисляются **ДО** присваиваний
2. `[%var="значение"]` в ответе не доступно в `[if(%var)]` того же шага
3. `$entity` — доступна сразу; `%var` (старое значение) — доступен сразу

### Navigation stack
`(семантика → §6, детали → B_RUNTIME_MODEL.md §4)`
- `[@goto]` — стек сбрасывается. Пользователь в целевом контексте.
- `[@extend]` — стек сохраняется. Возврат в исходный узел.

### Slot filling runtime
`(описание → §5, детали → B_RUNTIME_MODEL.md §6)`
- Автономный цикл, не зависящий от дерева узлов
- При `retry_count === 0` → `failure_answer` → fallback

### Dictionary resolution
- `[dict(name)]` в правилах: матчинг по всем строкам словаря (канон + синонимы)
- `[dict(name)]` в ответах: case-insensitive подстановка
- `[&dict(name, norm)]` в ответах: подстановка нормализованного канона

### Service tags runtime
`(полное описание → B_RUNTIME_MODEL.md §7)`
- `[nothing]` — без него узел с пустой репликой пропускается
- `[disableautovars]` — продляет жизнь `%that_anchor` / `%last_mark` на текущем шаге
