# E. СТРОКОТИПИЗИРОВАННАЯ СИСТЕМА ФОРМАТИРОВАНИЯ ОТВЕТОВ (RESPONSE_FORMATTING)

**SSOT для:** Формальное исполнение строки ответа, 3-слойная архитектура (FORMAT/TEMPLATE/CONTROL_FLOW), lifecycle udict/sudict, namespace-модель переменных, placeholder-система, модификаторы выдачи, SSML,Usage Contract для навигации.

**НЕ содержит:** DSL-синтаксис/функции/Action (ссылка → A), Runtime семантику переходов (ссылка → B), Архитектуру узлов (ссылка → C), Правила генерации артефактов (ссылка → D).

---

## 1. Формальная модель исполнения строки ответа

Каждая строка в `answers` проходит через три независимых слоя в строгом порядке:

```
TEMPLATE_LAYER (Binding) → CONTROL_FLOW_LAYER (Logic) → FORMAT_LAYER (UI)
         (1)                         (2)                        (3)
```

### 1.1 Execution Order

| Порядок | Слой                  | Операции                                                 | Результат                              |
|---------|-----------------------|-----------------------------------------------------------|----------------------------------------|
| 1       | TEMPLATE (Binding)    | `%var` интерполяция, `[dict/udict/sudict]`, `[&N]`    | Raw string с подставленными значениями |
| 2       | CONTROL_FLOW (Logic)  | `[if/elsif/else]`, `--switch/--case`                    | Отфильтрованные text fragments        |
| 3       | FORMAT (UI Render)    | `[b/i/u/br/ul/ol/li/el/href/inf]`, модификаторы         | Final response payload               |

**Инвариант:** Ни один тег не может выполнять операции из двух слоёв одновременно.

### 1.2 Formal String Execution Pipeline

Пример трансформации строки:
```
Input:  :lu: [b][if(%city)]{Доставка [%city]}[/b]
```

| Шаг | Операция                        | Результат                                    |
|-----|---------------------------------|----------------------------------------------|
| 1   | TEMPLATE: `%city` → "Москва"    | `:lu: [b][if(%city)]{Доставка Москва}[/b]`  |
| 2   | CONTROL_FLOW: `%city` truthy    | `:lu: [b]Доставка Москва[/b]`                |
| 3   | FORMAT: `[b]`, `:lu:`           | Bold: "Доставка Москва" (once-per-cycle)     |

## 2. TEMPLATE LAYER (Данные и привязка)

Слой ответственен исключительно за подстановку данных в строку ответа. НЕ содержит логики и форматирования.

### 2.1 Переменные

| Конструкция          | Операция      | Слой  |
|----------------------|---------------|-------|
| `[%var]`        | Вывод значения     | TEMPLATE |
| `[%var="value"]` | Присвоение     | TEMPLATE |
| `[%var=""]`    | Сброс значения     | TEMPLATE |

`( синтаксис и ограничения имён см. A_LANGUAGE_SPEC.md §6 )`

### 2.2 Namespace Model

Переменные разделены на пространства имён для предотвращения коллизий:

| Namespace   | Префикс/Паттерн        | Жизненный цикл        | Пример                          |
|-------------|------------------------|-----------------------|---------------------------------|
| **user**    | `%user_*`         | Сессия (явное создание) | `%user_name`, `%user_phone`  |
| **voice**   | `%voice.*`        | Сессия (системные)    | `%voice.end_call`              |
| **system**  | `%auto_*`, `%reply-*`, `%query-*` | Runtime (движок)     | `%auto_reply_counter`          |
| **context** | `%that_anchor`, `%last_mark` | Шаг/Сессия           | `%that_anchor="confirm"`       |
| **action**  | `[[field]]`         | Action-call scope     | `[[status]]` внутри `{...}`    |

**Pравило namespace:** Переменные без точки (`.`) или нижнего подчёркивания префикса (`_*`) живут на уровне сессии. Переменные без точек и префиксов (`%var`) — свободный регистр автора, lifecycle = сессия.

### 2.3 Placeholder System

Абсолютно неопределённые плейсхолдеры (время разработки) оформляются строго:

```
[Здесь будет [действие: перевод на оператора]]
```

`( синтаксис вложенных `[]` допустим только внутри плейсхолдеров-заглушек )`

### 2.4 Дictionaries, udict, sudict

| Конструкция          | Поведение                          | Lifecycle сброса         |
|----------------------|-------------------------------------|---------------------------|
| `[dict(name)]`  | Случайная строка из словаря          | Каждый вызов (no state)  |
| `[udict(name)]` | Случайная строка, одна за цикл      | Новый шаг/шаг пользователя |
| `[sudict(name)]` | Случайная строка, одна за разговор   | Только новый диалог       |

**Formal Lifecycle Definition:**
- **Cycle State (udict):** Сбрасывается при начале нового шага пользователя. После прохода `udict` сохраняет случайно выбранную строку в скрытом регистре; повторные вызовы того же `udict` в рамках одного ответа пользователя вернут ту же строку.
- **Conversation State (sudict):** Сохраняется до конца сессии. После первого вызова `sudict(name)` строка зафиксирована для этого диалога.

### 2.5 References

```
[&1]
[&dict(city, norm)]
```

`( синтаксис и номерация см. A_LANGUAGE_SPEC.md §5 )`

### 2.6 Action Placeholders

```
[action(name, param="value")]{[[field]]}
```

Результат action доступен только внутри `{...}` через `[[field]]`.

`( синтаксис action см. A_LANGUAGE_SPEC.md §11 )`

## 3. CONTROL_FLOW LAYER (Логика выполнения)

Слой ответственен исключительно за фильтрацию текста. НЕ содержит форматирования и НЕ содержит присвоений в составе форматированных блоков.

### 3.1 Единый стиль: Inline IF

Проект использует **исключительно inline IF**. Block IF (`--if`) разрешён только для `--switch/--case/--endswitch` (см. 3.2).

```
[if($entity)]{Текст}
[if(%var)]{Текст}
[if(!%var)]{Текст}
[if(%var="value")]{Текст}

[if(cond)]{A} [elsif(cond2)]{B} [else]{C}
```

**Правила Inline IF:**
- Закрывающая `}` — в КОНЦЕ всего блока.
- Без пробела между `)` и `{`.
- В `{...}` может быть: текст, присвоения, навигация, форматирование.
- `[if]` **НЕ** может находиться внутри тега форматирования.

### 3.2 Switch/Case (Только блочный)

Switch используется только для мультипликации веток:

```
--switch([%var1])
--case(1)
Ответ для 1
--case(2)
Ответ для 2
--default
Дефолтный ответ
--endswitch
```

Каждый тег switch/case/endswitch — **отдельный block** в `answers.blocks`.

### 3.3 Навигация: Usage Contract

| Конструкция                     | Usage Contract в ответе                        | Runtime семантика           |
|---------------------------------|------------------------------------------------|-----------------------------|
| `[@goto("Node")]`       | Перенаправление потока на узел                | Ссылка → B_RUNTIME_MODEL.md §4 |
| `[@extend("Node")]`      | Вставка ответа узла с возвратом               | Ссылка → B_RUNTIME_MODEL.md §4 |
| `[ExtendAnswer("Global")]`| Подключение глобального ответа                 | Комpozитный узел (Ссылка → C §1) |

**Важно:** Навигация выполняется ДО форматирования. `[@goto]` может прервать рендер текущей строки.

### 3.4 Пустой ответ

```
[%var="1"] [nothing]
```

`( runtime behavior `[nothing]` см. B_RUNTIME_MODEL.md §7 )`

### 3.5 Служебные теги

| Тег                      | Назначение                           | Reference              |
|--------------------------|---------------------------------------|------------------------|
| `[disableautovars]` | Продление жизни якоря/метки          | B_RUNTIME_MODEL.md §7 |

## 4. FORMAT LAYER (UI Rendering)

Слой ответственен исключительно за внешний вид ответа. НЕ содержит логики и НЕ содержит подстановок данных (данные уже подставлены слоями 1 и 2).

### 4.1 Formatting Tags

| Тег                                | Назначение         |
|------------------------------------|--------------------|
| `[b]...[/b]`                  | Жирный             |
| `[i]...[/i]`                   | Курсив             |
| `[u]...[/u]`                   | Подчёркнутый       |
| `[br]`                       | Перенос строки      |
| `[ul]...[li]...[el]`          | Ненумерованный список |
| `[ol]...[li]...[el]`          | Нумерованный список    |
| `[href(url)([_blank])]{текст}` | Ссылка             |
| `[inf("скрытый")]{видимый}`    | Юзерлинк            |

**Pравило:** Форматирование — это обёртка. Внутри `[b]...[/b]` может быть текст и переменная, но НЕ `[if]` и НЕ `--switch`.

### 4.2 Модификаторы выдачи

Модификатор в начале блока/строки (перед текстом и форматированием). Допустимо комбинирование.

| Модификатор  | Поведение                        |
|--------------|----------------------------------|
| `:u:`    | Один раз за сессию               |
| `:lu:`   | Один раз за цикл; сброс на новом |
| `:last:` | Пониженный приоритет             |
| `:+n:`    | Сдвиг веса (повышение)          |
| `:-n:`    | Сдвиг веса (понижение)          |

**Синтаксис:** без скобок, с пробелом после модификатора: `:lu: Текст`.

### 4.3 SSML в голосовом ответе

```
<speak>Рада была помочь.<break time='1500ms'></speak> [%voice.end_call="True"]
```

SSML обрабатывается движком как raw-строка (проходит сл слои 1 и 2).

## 5. Эталонные паттерны (Usage Examples)

### 5.1 Переменные и присвоения
```
Доставка [%pizza_type] [%pizza_size]
Я запомню, что вы [&1] [%user_name="[&1]"]
Хорошо. [%user_name=""]
Извините, не расслышала. Повторите. [disableautovars]
До свидания. [%voice.end_call="True"]
```

### 5.2 Словари
```
:lu: [b][dict(hello_ans)]![/b]
[udict(anec)]
```

### 5.3 Списки и юзерлинки
```
Что любишь:
[ul] [inf("Чай")]{чай} [li] [inf]{кофе} [el] [%last_mark="чай или кофе?"]
```

### 5.4 Референции и таблицы
```
Мы в [&dict(messenger, norm)]:
[br] [@hvs("messengers", "[&dict(messenger, norm)]")]
```

### 5.5 Action
```
[action(weather, lon_lat="[&dict(1, norm)]")]{[%status="[[status]]"]}
```

### 5.6 Навигация
```
[@goto("Узел 2.1 - Начало дерева")]
[ExtendAnswer("Глобальный ответ")]
```

## 6. MUST NOT (Strict Layer Violations)

### 6.1 Смешивание слоёв

| Violation                      | Нарушение                        | Правило                     |
|--------------------------------|----------------------------------|-----------------------------|
| `[b][if($x)]{...}[/b]`   | IF внутри FORMAT                   | CONTROL_FLOW → FORMAT, не наоборот |
| TEMPLATE внутри FORMAT внутри CONTROL_FLOW | Данные в форматированном блоке логики | Разделить слои |
| `action + assignment + dict` в одной конструкции | Три слоя в одном выражении   | Последовательное выполнение |
| `--if` / `--elsif` / `--else` в одной строке | Блоковый IF вместо inline  | Только inline IF           |

### 6.2 Общие запреты

| № | Запрет                                        |
|---|-----------------------------------------------|
| 1 | ❌ HTML `<b>`, `<br>` → использовать `[b]`, `[br]` |
| 2 | ❌ backticks, `id="..."` в ответах            |
| 3 | ❌ Все ответы в один inline-словарь           |
| 4 | ❌ Закрывающая `}` не в конце блока if        |
| 5 | ❌ `$` для переменных — только `%`            |
| 6 | ❌ `== True`, `>=`, `++`, `--` в условиях. ❌ `@@`, `CRM_Trans:`. ❌ `$$prev` |
| 7 | ❌ `$var=value` ✅ `[%var="[$entity]"]`   |
| 8 | ❌ Смешивание задавателя и обработчика (Ссылка → C §3) |
| 9 | ❌ Технические комментарии в DL-ответы      |
