# Проект: Генератор чат-ботов для ДОС

## 1. Структура репозитория

```
LLM/
├── docs/                 # База знаний для агентов (DSL, правила, архитектура)
├── agents/       # Промпты агентов конвейера
├── work/                 # Результаты агентов (один бот = подпапка, см. work/README.md)
│   └── <BotSlug>/        # 2_ARCHITECT__MAP, 3_CREATIVE__PHRASES, 9_PACKAGER__STAGING, output/nodes
├── archives/exported/    # Готовые ZIP для загрузки в ДОС
├── scripts/              # prepare_staging, validate_archive, pack_archive (не агенты)
├── .opencode/            # OpenCode в VS Code
└── opencode.json
```

## 2. Конвейер агентов

См. `agents/1_ORCHESTRATOR.md`: Архитектор → Креативщик → DL Правила → ML → Копирайтер → DL Ответы → Валидатор → **скрипты** (`9_PACKAGER.md`).

Артефакты — в `work/<BotSlug>/`. Итоговый ZIP — в `archives/exported/`. Завершённые боты не оставляют 9_PACKAGER__STAGING в `work/` (только ZIP + `FACTORY_PROJECTS_MEMORY.md`).

## 3. Управление памятью

| Файл | Назначение |
|------|------------|
| `PROJECT_MEMORY.md` (этот файл) | Как устроена фабрика, этапы, правила репозитория |
| `FACTORY_PROJECTS_MEMORY.md` | Конкретные боты в работе: на каком узле и у какого агента остановились, чтобы возобновить генерацию |

Оркестратор обязан вести `PROJECT_MEMORY.md` (этот файл) и `FACTORY_PROJECTS_MEMORY.md` при каждом шаге и паузе. Подробности — в `agents/1_ORCHESTRATOR.md` и корневом `README.md`.

## 4. Технические снимки

> Краткая выжимка правил для быстрой ориентации. Полные спецификации находятся в `docs/`.

### Структура архива ДОС — кратко

| Сущность | Формат ID |
|----------|-----------|
| DialogNode, Dictionary, Skill, … | `8-4-4-4` (20 hex) |
| Assistant, Branch | полный UUID (32 hex) |

ZIP: `metadata.json` + папка `[ASSISTANT_ID]-master-[40 hex]` → сущности в подпапках.
Подробности: `docs/D_GENERATION_RULES.md`, `docs/F_PLATFORM_API.md`.

### Критические правила

1. Осмысленные названия навыков и узлов (не «Навык 1»).
2. ML: ≥10 уникальных фраз на класс, ≥3 класса.
3. `[&1]` в ответе требует `[-*-]` в правиле.
4. Канон словаря без `~` и `*`; синонимы через `=>`.
5. Слоты `required: true` → `retry_count` + `failure_answer`.
6. Переменные в ответах: `[if(%var)]` при риске пустого значения.
7. Блочные условия в ответах: только `--switch`, `--if` (двойной дефис).

## Этап 7 (май 2026): Агент Координатор

### Зачем
Накопилась путаница: неочевидно, в какой файл записывать новую информацию, какие файлы обновлять при изменении данных, как поддерживать перекрёстные ссылки.

### Что сделано
- Создан агент Координатор (`agents/11_COORDINATOR.md`) — советник Оркестратора по файловому устройству.
- Координатор знает всю структуру проекта: слои A–G, память, рабочие папки, промпты агентов.
- Координатор **только консультирует**, не вносит правки самостоятельно.
- Координатор **никогда не предлагает создавать новые файлы** — всегда находит место в существующих.
- Оркестратор обязан обращаться к Координатору перед любой записью информации от пользователя.
- Добавлен Координатор в таблицу агентов оркестратора и в список идентификации в чате.
- Обновлён формат идентификации: все агенты пишут `(Имя:)` перед каждым сообщением.

## Этап 8 (май 2026): Строготипизированная E_RESPONSE_FORMATTING (3-слойная модель)

### Зачем
`E_RESPONSE_FORMATTING.md` стал "сбродамищем": mixing FORMAT + CONTROL_FLOW + TEMPLATE в одном выражении, два стиля IF (inline и block),不明确 lifecycle udict/sudict, `[[status]]` placeholder без формальной модели, namespace `%voice.*` без определения.

### Что сделано
- **Разделение на 3 независимых слоя:** TEMPLATE (Binding) → CONTROL_FLOW (Logic) → FORMAT (Rendering). Строгий execution order.
- **Официальный invariant:** Ни один тег не может выполнять операции из двух слоёв одновременно.
- **Единый стиль IF:** Inline IF (`[if/elsif/else]`). Block IF (`--if`) полностью запрещён (в MUST NOT).
- **Namespace model:** `user_*`, `voice.*`, `system_*`, `context`, `action` — формально определены.
- **Placeholder system:** `[Здесь будет [действие: ...]]` официально задокументирован.
- **Dictionary lifecycle:** udict (cycle-state), sudict (session-state) формализованы в E §2.4 и B §7.
- **Runtime семантика `@goto`/`@extend`:** перенесена в B; в E只剩 usage contract.
- **Cross-references:** `F_PLATFORM_API.md`, `8_VALIDATOR.md`, `7_RESPONSES_AUTHOR.md`, `11_COORDINATOR.md` обновлены.
- **Audit:** 0 фактов потеряно, 0 дублей, 0 прорех.
