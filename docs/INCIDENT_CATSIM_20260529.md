# Критические инциденты: cat_simulator (2026-05-29)

## Сводка
Три критические ошибки, помешавшие обучению бота в ДОС. Все три — результат недостаточной валидации на этапе сборки и пробелов в промптах агентов.

---

## Инцидент 1: Переменная cat_mood без @@set

**Проявление:** В ответах (`6_RESPONSES_AUTHOR__RESPONSES.md`) массово использовалась переменная `%cat_mood` для выбора реплики по настроению. Однако ни в одном узле отсутствовала команда `@@set` для инициализации `cat_mood`. Переменная существовала только как значение в `[if(%cat_mood="content")]`, но никогда не присваивалась.

**Следствие:** В ДОС переменная `cat_mood` всегда `null`. Все `[if(%cat_mood="...")]` дают false. Бот не произносит ни одного ответа — мёртвый узел.

**Причина:**
- Архитектор определил `%cat_mood` как переменную в карте, но не прописал, где и когда она должна инициализироваться.
- DL Автор Ответов использовал `%cat_mood` в `[if()]`, не проверив, где она задётся.
- Скрипт `generate_all_nodes.py` не выводит предупреждений при использовании `%var` без присвоения.
- Валидатор (`8_VALIDATOR.md`) проверяет переменные, но не инициализацию.

**Урок:**
> **Любая переменная, используемая в `[if(%var)]` или `[%var="value"]`, ДОЛЖНА иметь хотя бы одно место инициализации (`@@set %var=...`) в каком-либо узле.** Если переменной нет в секции "Условия:" ни одного узла с присвоением — она будет `null` на платформе.

**Исправление:** Добавить `@@set` в узел "Загрузка" или корневой узел:
```
## Узел: Загрузка
EVENT 00b2fcbe-f27f-437b-a0d5-91072d840ed3 *

## Узел: Мяу! Что будем делать?
Ответы:
:lu: @@set %cat_mood="content" [@goto("Мяу! Что будем делать?")]
:lu: @@set %cat_mood="hungry" [@goto("Мяу! Что будем делать?")]
...
```

## Инцидент 2: Правила в разделе ответов

**Проявление:** В `3_RULES_AUTHOR__RULES_AND_DICTIONARIES.md` правила узла (строки с `*`, `[@cmb]`, `EVENT`) написаны в формате `## Узел: Имя\n* правило *`, что является правильным для файла правил. Однако `generate_all_nodes.py` ожидает, что файл правил содержит ТОЛЬКО правила, а файл ответов — ТОЛЬКО ответы. Часть правил оказалась продублирована в `6_RESPONSES_AUTHOR__RESPONSES.md` в секции "Ответы:", что привело к их попаданию в поле `responses` вместо `dl_rules` в JSON.

**Следствие:** Правила матчинга оказались в поле `responses` JSON-узла, где они не обрабатываются движком DL. Узел не распознаёт пользовательские фразы.

**Причина:**
- DL Автор Ответов добавил блок `Правила:` в файл ответов, нарушая строгое разделение ответственности.
- Валидатор не проверяет наличие DL-синтаксиса (`*`, `[@cmb]`, `EVENT`) в файле ответов.

**Урок:**
> **Файл `6_RESPONSES_AUTHOR__RESPONSES.md` содержит ТОЛЬКО секции "Условия:" и "Ответы:". Любая строка с `*`, `[@cmb]`, `EVENT` в файле ответов — критическая ошибка.**

## Инцидент 3: --include словари не извлечены

**Проявление:** Словарь `hello_all` использует `--include(hello_hello)`, `--include(hello_zdravstvuy)`, `--include(hello_helloenglish)`. Словарь `hello_hello` использует `--include(hello_dobr_utro)` и `--include(hello_dobr_vecher)`. Словари `hello_dobr_utro` и `hello_dobr_vecher` существуют в `3_RULES_AUTHOR__RULES_AND_DICTIONARIES.md`, но `build_full_staging.py` не рекурсивно извлекает `--include`.

**Следствие:** `hello_dobr_utro` и `hello_dobr_vecher` не созданы как отдельные Dictionary JSON в staging. `hello_hello` ссылается на несуществующие словари — матчинг сломан.

**Причина:**
- `build_full_staging.py` обрабатывает `--include` поверхностно: не рекурсивно и не создаёт отдельные сущности для includенных словарей.
- Валидатор не проверяет, что все `--include` имеют соответствующие Dictionary JSON.

**Урок:**
> **Все словари, referenced через `--include()`, ДОЛЖНЫ быть извлечены как отдельные Dictionary JSON-файлы. Сборщик обязан рекурсивно разрешать includes.**
