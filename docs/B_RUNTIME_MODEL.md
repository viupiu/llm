# B. МОДЕЛЬ ИСПОЛНЕНИЯ (RUNTIME_MODEL)

**SSOT для:** Порядок исполнения, lifecycle переменных, поведение IC, семантика `@goto`/`@extend`, поведение slot filling, системные переменные.

**НЕ содержит:** Синтаксис (ссылка → A), Архитектуру бота (ссылка → C), API платформы (ссылка → F).

---

## 1. Execution Pipeline

### Порядок шага
1. Сбор сущностей из запроса
2. Вычисление условий (`conditions` и `[if]` в ответе)
3. Выбор ветки
4. Формирование ответа (`answers`)
5. Присваивания внутри ответа

**Критическое ограничение:** Движок **не** обновляет состояние «на лету» внутри шага. Каждый `[if]` в ответе обрабатывается **независимо**; `[else]` не привязан к соседним `if`. Изменения в `answers` не пересчитывают уже выбранные условия узла.

### Ограничение same-step assignment
Переменная, записанная на текущем шаге `[%city="Москва"]`, **НЕ** доступна в условиях этого шага `[if(%city)]`, потому что условия вычисляются ДО генерации ответа.

### Доступность значений

| Тип | Доступность в условиях текущего шага |
|-----|--------------------------------------|
| `$entity` | сразу |
| `%var` (старое) | да |
| `%var="новое"` в ответе | только со следующего шага |
| `%_var_` (временная) | сразу, если задана до проверки |

Сущности (`$city`) живут только в рамках текущей реплики; для переноса — `[%city="[$city]"]`.

### Timing вычислений
Движок собирает все присваивания ответа и применяет их после генерации всей реплики. Временные переменные `%_var_` доступны сразу внутри того же шага, если они заданы ранее в ответе (используются внутри `[if(...)]`).

## 2. Runtime State

### Системные переменные
Заполняются движком; в ответах — `[%имя]`, в условиях — `%имя`.

| Переменная | Назначение | Примечание |
|------------|------------|------------|
| `%reply-` | Текст предыдущего ответа бота | Voice/chat; SSML читается как текст |
| `%reply--` | Предпредыдущий ответ | Редко |
| `%query-` | Текущий запрос пользователя | Отладка, эхо |
| `%query--` | Предыдущий запрос | Контекстные уточнения |
| `%auto_reply_counter` | Счётчик ответов из одного ДУ | Эскалация (3 раза → оператор) |
| `%auto_request_counter` | Счётчик запросов подряд | Лимит «не понял» |
| `%auto_request_length` | Длина запроса в словах | «Слишком длинный запрос» |
| `%user_timezone` | Часовой пояс | Для `[@Date]`, `[@Time]` |
| `%wflag_hide_reply` | Скрыть реплику (`"1"`) | Web/chat; не для голоса |
| `%voice.end_call` | Завершить звонок (`"True"`) | Телефония |

### Счётчики
Счётчики увеличиваются явным присваиванием в ответе:
`(синтаксис см. A_LANGUAGE_SPEC.md)`
```dl
[if(!%cnt)] Не понял. [%cnt="1"]
[if(%cnt="1")] Не понял. [%cnt="2"]
```

## 3. Matching Runtime

### Preprocessing
Движок нормализует запрос пользователя **ПЕРЕД** матчингом:

| Преобразование | Пример |
|---|---|
| `ё` → `е` | `ёж` → `еж` |
| Тире → пробел | `Санкт-Петербург` → `Санкт Петербург` |
| Схлопывание пробелов | `  много   пробелов` → `много пробелов` |

**Следствие:** в правилах пишите `е` вместо `ё`. Канон в словарях — грамотно (`Санкт-Петербург`), но матчинг произойдёт по нормализованному запросу.

### Case normalization
Движок нормализует запрос к нижнему регистру перед матчингом. Достаточно одного варианта: `Александр` или `александр` — движок поймёт оба.

### Унисекс-имена
Имена вроде `Саша`, `Женя` должны присутствовать в обоих словарях (муж./жен.), если бот различает пол.

### Entity lifecycle
Сущности извлекаются на шаге 1 Execution Pipeline (`$entity`). Они доступны в условиях текущего шага, но не сохраняются автоматически на следующий шаг. Для сохранения необходимо явное присваивание: `[%var="[$entity]"]`.

## 4. Navigation Runtime

### Семантика `@goto` и `@extend`
`(синтаксис см. A_LANGUAGE_SPEC.md)`

| | `@goto` | `@extend` |
|---|---------|-----------|
| Возврат в исходный узел | нет | да |
| Контекст пользователя | целевой | исходный |
| Назначение | смена состояния | переиспользование текста |

**Запрещены циклы:** `A → B → A` или цепочки extend без выхода.

### Таблица навигационных сценариев

| Сценарий | Исходный ответ | Целевой ответ | Итог | Пользователь в |
|---|---|---|---|---|
| goto | `Привет! [@goto("2")] Как дела?` | `Мой друг!` | `Привет! Мой друг!` | целевом |
| goto + continue | то же | `Мой друг! [continue]` | `Привет! Мой друг! Как дела?` | целевом |
| goto + break | то же | `Мой друг! [break]` | `Привет! Как дела?` | исходном |
| extend | `Привет! [@extend("2")] Как дела?` | `Мой друг!` | `Привет! Мой друг! Как дела?` | исходном |
| extend + break | то же | `Мой друг! [break]` | `Привет! Как дела?` | исходном |

## 5. IC Runtime Model (Intent Classification)

### Confidence and Thresholds
- Каждый интент имеет **индивидуальный трешхолд**. Если `confidence` ниже порога — ответ не выдаётся, поток уходит в fallback.
- **F1-мера:** критический порог `0.7`.
- Если в узле заполнены примеры, система вычисляет confidence score. Если выше порога и выше, чем у подходящих правил (Rules) — узел IC сработал.

### Semantic Conflicts
Схожие примеры в разных узлах снижают уверенность и повышают трешхолды. Движок генерирует матрицу конфликтов для диагностики. Если интенты отличаются только одним аспектом (например, тип товара) — объединить в один узел + слот-филинг. Это снижает конфликты и повышает F1.

### Dataset Constraints
- ≥ 10 уникальных примеров на класс
- ≥ 3 разных класса в боте
- Рекомендуется ≥ 100 примеров на класс
- Балансировка: примеры должны покрывать вариативность формулировок (синонимы, порядок слов, падежи)

### Interaction IC vs Rules
При срабатывании по IC движок игнорирует блок правил. Референции `[&1]` не сработают. **Не использовать референции в ответах узлов IC.** Правила работают по жестким паттернам. IC — по семантике.

## 6. Slot Filling Runtime

`(синтаксис см. A_LANGUAGE_SPEC.md)`

- Не блокирует дерево: запрашивает слоты и уводит в fallback при неудаче.
- Повторный ввод не гарантирует сохранение контекста — явно пишите переменные.
- Параметры конфигурации: `variable`, `entity`, `question`, `retry_count`, `failure_answer`, `required`.

## 7. Service Runtime Tags

### Пустой ответ
Для намеренно пустой реплики используется тег `[nothing]`. Без него движок пропускает узел с пустым ответом.
`(синтаксис см. A_LANGUAGE_SPEC.md)`

### Dictionary Lifecycle

| Конструкция     | Scope | Trigger for reset |
|-----------------|-------|-------------------|
| `[dict(name)]`  | Stateless | Каждый вызов — новый рандом |
| `[udict(name)]` | Cycle | Новый шаг пользователя сбрасывает счётчик |
| `[sudict(name)]` | Session | Новый диалог сбрасывает счётчик |

Engine stores the hash of the last selected string per dictionary name. For `sudict`, the hash persists in session state. For `udict`, the hash is stored in step-local state and discarded at the next user turn.

### Service Tags

| Тег | Runtime behavior |
|---|---|
| `[disableautovars]` | Продлить жизнь якоря/метки |
| `[nothing]` | Намеренно пустая реплика |
| `[%voice.end_call="True"]` | Завершение звонка (runtime effect) |

### Anti-repeat mechanism

**MUST:** Для предотвращения повторов используйте `:lu:` (limit usage) или `[udict(name)]` — они обеспечивают автоматическое перебирание элементов словаря без повторов в рамках цикла/сессии.

**MUST NOT:** Не использовать ручные счётчики (`%excuse_count`, `%last_excuse_style` и т.п.) для anti-repeat. Это избыточно и усложняет архитектуру.

**Инцидент (2026-05-29, Excuse Gen Bot):** Архитектор предложил `%excuse_count` + `%last_excuse_style` для защиты от повторов. Правильный подход — встроенные механизмы `:lu:` / `[udict()]`.
