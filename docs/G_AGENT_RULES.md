### Границы ответственности (Архитектор)
Архитектор проектирует логику, переходы и интенты. Архитектор **НЕ ИМЕЕТ ПРАВА** писать DL-правила, "скелеты правил", конструкции вида `* [@cmb({...})] *`, и навязывать конкретные словари. Архитектор может рекомендовать словари по описанию ("в узле X по смыслу может подойти `dict(Y)`"), но итоговое решение принимает **Автор Правил**.

**Критические ошибки Архитектора (запрещены):**
1. **Генерация "скелетов правил"** — строки вида `* [@cmb({...})] *`, `* [dict(...)] *`. Архитектор не умеет писать правила, и такие строки вводят Автор Правил в заблуждение.
2. **Склеивание интентов** — объединение нескольких семантически разных паттернов в одно правило (например, `[dict(help)]`, `[dict(common_how)]`, `[dict(common_what)]` в одном `[@cmb]`).
3. **Навязывание неподходящих словарей** — использование словарей, содержание которых не соответствует смыслу узла (например, `keywords_company` содержит юридическое значение "компания", а не "друзья/компания для просмотра"). Архитектор судит о словарях только по описанию и не видит их содержимое.
4. **Неверная интерпретация `beginning_words`** — словарь `beginning_words` содержит междометия и вводные слова (НЕ приветствия). Используется только как факультативный `<[dict(beginning_words)]>` в начале правила с `[@cmb_s()]`, в конце — `<[dict(anysign)]>`.

### Кодировка и допустимые символы

- ✅ **Кириллица** — в русских словах. Все буквы русского алфавита (а–я, ё) — корректны.
- ✅ **Латиница** — в иностранных словах, технических терминах, именах файлов, узлах, переменных.
- ✅ **Эмодзи и спецсимволы** — допустимы как элементы дизайна оформления (смайлики, маркеры, разделители).
- ❌ **Кракозябры / битые символы** вместо букв — недопустимы. Любой символ, который является искажением русской или латинской буквы (например, `��`, `â`, `Ã`, `ð` вместо нормальной буквы) — это ошибка кодировки. Агент обязан проверять генерацию на отсутствие подобных артефактов.

## 5. Генерация пользовательских фраз (Креативщик)

### 5.1. Базовые требования
- ❌ Генерация грамматически неверных фраз, калек с английского (напр. "Зовется [Имя]", "Со мной можешь обращаться [ИМя]").
- ✅ Строгая проверка естественности русского языка. Любая фраза должна звучать так, как реально говорит носитель. Если сомневаетесь — уберите её.

### 5.2. Anti-Pattern: Pseudo-Diversity (Псевдо-разнообразие)

**Определение:** Pseudo-diversity — это генерация фраз, которые отличаются друг от друга только 1–2 словами при сохранении одинакового синтаксического скелета (sentence skeleton). Это НЕ разнообразие. Это lexical permutation, template exhaustion, cosmetic variation.

**Типичные примеры pseudo-diversity:**
```
BAD:
Всё, достаточно, думаю, хватит
Всё, достаточно, думаю, хватит на сегодня
Всё, достаточно, думаю, хватит на этот раз
Думаю, хватит пока
Наверное, хватит пока

BAD:
Хочу фильм
Хочу посмотреть фильм
Хочу посмотреть какой-нибудь фильм
Хочу посмотреть какой-нибудь хороший фильм
```

**Почему это критично:**
- ML-классификатор обучается на таких данных и не видит реального распределения пользовательской речи.
- DL-правила не покрывают реальные жизненные сценарии.
- Набор фраз выглядит большим, но семантически представляет 1–2 шаблона.

**Критерии FAILED генерации:**
1. **Sentence Skeleton Repetition:** 3+ фраз подряд имеют одинаковый синтаксический скелет (подлежащее + сказуемое + дополнение в том же порядке, с заменой только синонимов/уточнений).
2. **Lexical Loop:** >20% фраз узла начинаются или заканчиваются одинаковыми стоп-словами в той же последовательности.
3. **Cosmetic Variation Only:** >10% фраз отличаются только добавлением/удалением 1–2 слов (например, `спасибо` → `спасибо большое` → `очень спасибо`).

**Правильный подход — Contextual Expansion:**
Когда короткие варианты исчерпаны, Креативщик ОБЯЗАН расширять контекст:
- Причины: `Ладно, потом продолжу, сейчас обедать пора`
- Эмоции: `Ну всё, хватит на сегодня, голова уже не варит`
- Предыстория: `Спасибо, я уже достаточно набрался информации по этой теме`
- Сомнения: `Мне кажется, пока хватит, а то дальше только запутаюсь`
- Бытовые детали: `Лучше остановимся, нужно собираться на встречу`

**Обязательные оси варьирования (6 осей):**
1. **Эмоция** — спокойствие, раздражение, усталость, срочность, сомнение, радость, неловкость
2. **Контекст** — дома, на работе, в дороге, ночью, после ошибки, после разговора с оператором
3. **Уровень уверенности** — уверенный, неуверенный, полу-вопрос, размышление вслух
4. **Длина** — ультракороткие, средние, длинные, поток мысли
5. **Стиль речи** — разговорный, деловой, нервный, небрежный, вежливый, обрывочный
6. **Noise** — опечатки, лишние слова, самоисправления, повторения, сбивчивость

**Распределение по длинам (обязательно):**
- 10–15%: ультракороткие (1–3 слова)
- 30–35%: средние (4–8 слов)
- 30–35%: длинные контекстные (9+ слов, с причинами, эмоциями, предысторией)
- 10–15%: поток мысли / messy input (с самоисправлениями, сбивчивостью, шумом)

### 5.3. Примеры Good vs Bad

**BAD (Pseudo-Diversity):**
```
Спасибо
Спасибо большое
Очень спасибо
Большое спасибо
Спасибо, вам большое
```

**GOOD (Contextual Expansion):**
```
Спасибо
Большое спасибо, это именно то, что я искал
Ок, всё понял, спасибо за помощь
Ну и лады, спасибо, буду знать
Всё, спасибо, если что ещё нужно — обращусь
Спасибо, этого более чем достаточно для принятия решения
Ладно, спасибо, дальше уже сам разберусь
Думаю, информации пока достаточно, не хочу перегружаться
На сегодня, наверное, закончим, я уже устал читать
Мне кажется, этого уже хватит, дальше только запутаюсь
```

- Пустые `[- -]`.
- `=>` вне нормализующих словарей.

## 8. Методология проектирования DL-структур

**От анализа источника к сборке правила (эталон FPK):**

1. **Сбор формулировок:** извлечь все варианты фраз пользователя из ТЗ, скриптов и т.п.
2. **Кластеризация:** сгруппировать формулировки по интентам/узлам (одна реплика = один узел).
3. **Выделение ключевых слов:** определить смысловые элементы каждого интента.
4. **Скелет правила:** завернуть ключевые слова в `[@cmb({...})]`, оборачивая в `* {скелет} *`.
5. **Словари:** вынести повторяющиеся/многочисленные синонимы в `[dict(name)]`. В `3_RULES_AUTHOR__RULES_AND_DICTIONARIES.md` — явный заголовок `## Словарь: <имя>`, только стеммы.
6. **Вариативность:** покрыть падежи через `{кот/коту/котом}`, `~`, или `word_`-словари.
7. **Референции:** где нужно подставить слово из вопроса → `[-{...}-]` в правиле, `[&N]` в ответе.
8. **ML-примеры:** ≥10 на класс, вариативных, без конфликтов.

**Организация правил:**
- Дробление: один узел = одно намерение. Не смешивайте интенты.
- Сжатие: общие словари вместо дублирования лексикона в каждом узле.

## 10. FILE SAFETY RULES

- NEVER rewrite entire files unless explicitly instructed.
- ALWAYS prefer minimal diff patches.
- NEVER change file format.
- NEVER auto-convert markdown/yaml/txt into json.
- NEVER delete existing sections unless explicitly instructed.
- Preserve all unrelated content exactly.
- If file structure is unclear — STOP and ask.
- Before writing:
  1. Read file
  2. Analyze structure
  3. Modify only target section
  4. Validate syntax
  5. NEVER attempt autonomous recovery rewrites.
  6. If corruption detected: stop, explain, propose minimal fix, wait.
   7. Before ANY write operation: show planned changes, explain target files, why safe.

## 11. Коммиты и Git

- **MUST:** Текст коммита **ВСЕГДА ТОЛЬКО на русском языке**. Запрещено писать commit message на английском или любом другом языке.
- **MUST NOT:** Commit message на английском языке. Любое commit message, содержащее английские слова, является нарушением.
- В конце каждого commit message Оркестратор обязан добавлять суффикс ` (Оркестратор)` для отличия от коммитов пользователя.
- Формат: `{короткое описание изменений на русском} (Оркестратор)`.
- Пример: `обновлены DL-правила для узла greet_user (Оркестратор)`

**Инцидент (2026-05-28):** Оркестратор закоммитил изменения с английским commit message: `feat: add movie_rec_bot brief, new agents, scripts, and docs`. Нарушение правила §11. Commit был отменён через `git commit --amend` с русским сообщением. Урок: Оркестратор обязан формировать commit message исключительно на русском языке. Правило усилено до MUST NOT.

### Частота коммитов Оркестратора

- **По собственному усмотрению:** Оркестратор коммитит и пушит НЕ ЧАЩЕ 1 раза в час. Между авто-коммитами обязан выдержать интервал ≥60 минут.
- **По прямому запросу пользователя:** коммитить и пушить можно неограниченно, без ограничения по частоте.
- Если Оркестратор внёс изменения в файлы, но не получил запрос на коммит — он НЕ коммитит молча, а ожидает. Изменения остаются в unstaged до следующего авто-коммита (не чаще 1/час) или до явного запроса пользователя.
- Авто-коммит включает ВСЕ накопленные изменения (не только последние).

**Анти-паттерн:** Оркестратор совершает коммит после каждого редактирования файла. Это создаёт шум в git-логе. Коммиты — batch-операция, а не реакция на каждое действие.

## 12. REFACTOR GUARD — Обязательный протокол рефакторинга

REFACTOR GUARD — это НЕ отдельный агент, а обязательный режим выполнения любых операций редактирования текста, файлов, спецификаций и документов.

### 12.1. Когда включается

REFACTOR GUARD активируется автоматически при ЛЮБЫХ операциях:
- удаление дубликатов
- упрощение спецификаций
- реорганизация секций
- объединение правил
- реструктуризация документов
- дедупликация словарей, словарей-синонимов, правил, ответов
- переписывание артефактов агентов

### 12.2. Запрещённая стратегия

❌ **"обнаружил дубликат → удалить"**
❌ **"in-place deletion"** — прямое удаление без этапа канонизации
❌ **переписать без этапа canonicalization**
❌ **потерять структуру примеров**

### 12.3. Единственная разрешённая стратегия

✅ **"canonicalize → preserve → reconcile"**

Каждое изменение проходит четыре этапа:
1. **Canonical Version** — создаётся каноническая версия целевого документа/секции
2. **Diff Mapping** — выполняется сопоставление каждого элемента исходного текста с элементом канонической версии
3. **Safe Replacement** — выполняется безопасная замена только после прохождения всех проверок
4. **Audit Log** — формируется журнал изменений

### 12.4. Relocate Before Delete

Никакой элемент не удаляется сразу. Порядок:
1. Элемент переносится в canonical structure или merged section
2. Проверяется семантическая эквивалентность
3. Сохраняются все примеры, edge cases, предупреждения
4. Только после успешной проверки можно удалять оригинал

### 12.5. Semantic Equivalence Check

Перед объединением или удалением элемента:
- Проверить, что удаляемый элемент **действительно эквивалентен** целевому элементу
- Запрещено удалять "похожие, но не идентичные" правила
- Запрещено "схлопывать" правила с разными условиями, исключениями или границами

### 12.6. Preserve Examples

Все примеры из удаляемого блока:
- ДОЛЖНЫ быть сохранены
- Допускается только перенос, но не потеря
- Запрещено удалять примеры "как избыточные"

### 12.7. Preserve Edge Cases

Любые исключения, анти-паттерны и редкие условия:
- ДОЛЖНЫ быть сохранены отдельно
- Запрещено "схлопывать" edge cases в общие правила
- Каждый edge case должен сохранить свою уникальность и контекст

### 12.8. Preserve Warnings

Все формулировки MUST NOT / anti-pattern:
- Сохраняются без ослабления
- Нельзя заменять на более общие или мягкие формулировки
- Запрещено терять строгость предупреждений при рефакторинге

### 12.9. Audit Log — Обязательное логирование удалений

Каждый удалённый фрагмент ДОЛЖЕН быть явно зафиксирован:

| Поле | Содержание |
|------|-----------|
| **Что удалено** | Цитата или описание удаляемого фрагмента |
| **Куда перенесено** | Целевой файл, секция, строка |
| **С чем объединено** | Идентификатор канонического элемента |
| **Почему эквивалентность подтверждена** | Обоснование семантической идентичности |

### 12.10. Эффект

Система переходит от:
- **"упрощение через удаление"**

к:
- **"упрощение через структурную нормализацию"**

## 16. Библиотекарь (LIBRARIAN — REUSABLE KNOWLEDGE CURATOR)

**Цель:** Полный цикл поддержки `docs/reference/COMMON_DICTIONARIES.md` как canonical SSOT: ingestion из `raw_dicts/`, семантический анализ, deduplication, description maintenance.

### Границы ответственности (Библиотекарь)

#### Ingestion Pipeline
- ✅ **Dictionary ingestion**: импорт новых словарей из `raw_dicts/` в `COMMON_DICTIONARIES.md`
- ✅ **Duplicate detection (import)**: exact canonical name match перед импортом
- ✅ **Safe delete policy**: удаление из `raw_dicts/` ТОЛЬКО после успешного импорта
- ✅ **Import validation**: проверка JSON, парсинг, обработка ошибок
- ✅ **Import reporting**: структурированный отчёт (Imported / Skipped / Failed)
- ✅ **Registry consistency**: отсутствие дубликатов, орфанов, повреждённых ссылок

#### Semantic Curation
- ✅ Библиотекарь анализирует реальное содержимое словарей (стемы, не имена)
- ✅ Библиотекарь выявляет семантические дубликаты, misleading-имена, mixed-purpose словари
- ✅ Библиотекарь обновляет описания словарей в реестре на основе анализа
- ✅ Библиотекарь формирует индекс по интентам для удобства авторов правил

#### ЗАПРЕЩЕНО
- ❌ Библиотекарь удаляет строки из существующих словарей
- ❌ Библиотекарь удаляет словари из реестра
- ❌ Библиотекарь переименовывает словари (breaking change)
- ❌ Библиотекарь автоматически "очищает" дубликаты (destructive deduplication)
- ❌ Библиотекарь генерирует новые словари или DL-правила
- ❌ Библиотекарь перезаписывает существующие словари (silent replacement)
- ❌ Библиотекарь сливает словари без аудита и подтверждения (merge without audit)

### Границы ответственности (Оркестратор — Dictionary Pipeline)

**Оркестратор НЕ работает напрямую с:**
- ❌ `docs/reference/raw_dicts/`
- ❌ `docs/reference/COMMON_DICTIONARIES.md` (инпорт, обновление реестра)

## 17. Беклог — Active Governed Backlog (SSOT: `docs/беклог.md`)

`docs/беклог.md` — **active lifecycle-managed backlog**. Оркестратор обязан автоматически поддерживать его в актуальном состоянии **без напоминаний пользователя**.

### Правила для всех агентов
- ❌ **Inline TODO/FIXME/HACK** — Запрещено писать inline TODO/FIXME/HACK в коде и артефактах. Все задачи — в `docs/беклог.md`.
- ✅ **SSOT ссылки** — Если в файле необходимо указать на незавершённую работу, использовать короткую ссылку: `# TODO: A-2` или `→ docs/беклог.md`.
- ✅ **Новые задачи** — При обнаружении нового техдолга, незавершённой логики или улучшения — агент отчитывается Оркестратору; Оркестратор добавляет задачу в `docs/беклог.md` с уникальным ID, предварительно проверив absence semantic duplicates.
- ✅ **Завершённые задачи** — Агент отчитывается Оркестратору о завершении работы, чтобы Оркестратор мог выполнить Post-Task Backlog Check.

### Lifecycle States
| Status | Описание |
|--------|----------|
| `OPEN` | Новая задача, не начата |
| `IN_PROGRESS` | Работа над задачей начата |
| `PARTIALLY_RESOLVED` | Задача выполнена частично — записан progress note |
| `BLOCKED` | Зависит от другой задачи / внешнего условия |
| `RESOLVED` | Задача выполнена — перенесён в Архив |
| `ARCHIVED` | Итоговое состояние закрытой задачи |

### Semantic Completion Detection (Оркестратор)

Оркестратор обязан использовать **семантический анализ**, а не построчный поиск по ключевым словам, при проверке соответствия выполненной работы пунктам беклога.

**Примеры semantic match:**
- Backlog: "нужен lifecycle для словарей" + Выполнено: "добавлен LIBRARIAN" → **match**.
- Backlog: "пересмотреть описания словарей" + Выполнено: "LIBRARIAN обновил реестр" → **match**.
- Backlog: "Movie Rec Bot архитектурная карта" + Выполнено: "Архитектор переписал карту" → **match**.
- Backlog: "завершить упаковку FPK" + Выполнено: "pack_archive.py для FPK запущен успешно" → **match**.

**Алгоритм Post-Task Backlog Check:**
1. После завершения значимой задачи открыть `docs/беклог.md`.
2. Пройти по каждому `OPEN` / `IN_PROGRESS` / `PARTIALLY_RESOLVED` пункту.
3. Оценить семантическое перекрытие: покрывает ли выполненная работа этот пункт полностью, частично или не покрывает.
4. При полном перекрытии → `RESOLVED` → переместить в `## Архив` с датой и описанием.
5. При частичном перекрытии → `PARTIALLY_RESOLVED` → записать progress note, сократить формулировку до residual work.
6. Обновить `Last Update` всех затронутых пунктов.

**Duplicate Prevention (перед добавлением нового пункта):**
1. Проверить все active items (`OPEN`, `IN_PROGRESS`, `PARTIALLY_RESOLVED`).
2. Проверить semantic duplicates — задача с другим ID, но тем же смыслом.
3. Проверить slightly reworded duplicates.
4. Проверить overlapping tasks — задачи, которые частично перекрывают новую.
5. **ЗАПРЕЩЕНО** создавать дубликаты. Объединить с существующим или усилить существующий пункт.

### Refactor Guard для Беклога
- ✅ Preserve unresolved concerns, rationale, future risks, dependencies, architecture warnings.
- ❌ Удалять backlog item без semantic verification.
- ❌ Silently close tasks.
- ❌ Терять partially completed work.

### Backlog Memory Loop (Триггеры)

Перед **ЛЮБЫМИ** операциями Оркестратор обязан проверить беклог на:
1. Блокирующие unresolved задачи.
2. Conflicts с текущей операцией.
3. Planned future architecture changes.

Триггеры: рефакторинг, новый агент, новый runtime behavior, изменение DSL, изменение pipeline, запуск нового проекта, получение ТЗ.

### Координатор
- При проверке целостности конвейера и cross-references — включать `docs/беклог.md` в список проверяемых файлов.
- При обнаружении inline TODO/FIXME в артефактах — отчитаться Оркестратору для переноса в беклог.
