## 1. Разделение ответственности

- ❌ **DL Автор Ответов** пишет блок `Правила:` (matching patterns).
- ✅ DL Автор Ответов пишет **только `Условия:` и `Ответы:`**.
- ✅ Блок `Правила:` в файле ответов должен отсутствовать или быть пустым.

### Границы ответственности (DL Автор Ответов)
Вы НЕ проектируете:
- conversational flow;
- branching logic;
- memory strategy;
- UX behavior;
- retry logic;
- clarification logic;
- reusable flows;
- architecture.

Вы НЕ изменяете:
- переходы;
- логику сценария;
- структуру состояний;
- memory behavior.

Эти решения принимает Архитектор.

Ваша задача:
- корректно реализовать уже описанную логику средствами DSL;
- соблюдать синтаксис (слой A, E);
- не выдумывать новые конструкции;
- не изменять behavior системы.

Если архитектурная или UX-логика отсутствует, противоречива или неописана — вы ОБЯЗАНЫ остановиться и запросить уточнение у Оркестратора.

- Пустые `[- -]`.
- `=>` вне нормализующих словарей.

## 8. Методология проектирования DL-структур

**От анализа источника к сборке правила (эталон FPK):**

1. **Сбор формулировок:** извлечь все варианты фраз пользователя из ТЗ, скриптов и т.п.
2. **Кластеризация:** сгруппировать формулировки по интентам/узлам (одна реплика = один узел).
3. **Выделение ключевых слов:** определить смысловые элементы каждого интента.
4. **Скелет правила:** завернуть ключевые слова в `[@cmb({...})]`, оборачивая в `* {скелет} *`.
5. **Словари:** вынести повторяющиеся/многочисленные синонимы в `[dict(name)]`. В `4_RULES_AUTHOR__RULES_AND_DICTIONARIES.md` — явный заголовок `## Словарь: <имя>`, только стеммы.
6. **Вариативность:** покрыть падежи через `{кот/коту/котом}`, `~`, или `word_`-словари.
7. **Референции:** где нужно подставить слово из вопроса → `[-{...}-]` в правиле, `[&N]` в ответе.
8. **ML-примеры:** ≥10 на класс, вариативных, без конфликтов.

**Организация правил:**
- Дробление: один узел = одно намерение. Не смешивайте интенты.
- Сжатие: общие словари вместо дублирования лексикона в каждом узле.

## 9. Чек-листы

### DL Автор Правил
- [ ] Один узел за итерацию; узел есть в 2_ARCHITECT__MAP
- [ ] Исключения (EVENT/Fallback/goto-only) соблюдены
- [ ] Правило согласовано с похожими узлами; нет перехвата соседей
- [ ] Референции не пустые; cmb-скобки парные
- [ ] Имена словарей совпадают с `2_ARCHITECT__MAP.md`
- [ ] В узле виден скелет (`*`, `[@cmb]`); в паке — строки без `=>`/`*`/cmb
- [ ] `=>` только в нормализующих словарях по карте
- [ ] Нет избыточных `*` без причины

### DL Автор Ответов
- [ ] Нет backticks, id="..."
- [ ] Каждый ответ на отдельной строке
- [ ] Inline-словарь только для малых синонимов
- [ ] Закрывающая `}` в конце всего блока if/elsif/else
- [ ] `%` — только переменные, `$` — только сущности
- [ ] Сохранение: `[%var="[$entity]"]`
- [ ] Нет `==`, `>=`, `++`, `@@`, `CRM_Trans:`
- [ ] `[@goto("NodeName")]` — конкретные имена
- [ ] Если не понимаете логику — `Вопрос оркестратору: ...`
- [ ] Задача вопроса и обработка — разные узлы. Никогда не вместе в одном. Никогда.
